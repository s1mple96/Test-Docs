# BUG-【任通行、任货行】更换银行卡后未校验新银行协议勾选状态

## 【BUG描述】
用户在办理ETC绑定银行卡时,切换银行卡后,原银行协议的勾选状态被错误复用到新银行协议,导致用户未勾选新银行协议也能提交,存在法律合规风险。

## 【前置条件】
- 测试环境: 任通行/任货行小程序
- 测试页面: 办理ETC页面 (moduleApplyEtc/apply/bankCard)
- 测试数据:
  - 建设银行卡号: 6217XXXXXXXXXXXX
  - 光大银行卡号: 6226XXXXXXXXXXXX

## 【复现步骤】
1. 进入办理ETC页面
2. 输入建设银行卡号(6217开头)
3. 页面显示建设银行协议,手动勾选协议
4. 点击"下一步"按钮,提示"该卡不能使用"或其他错误
5. 清空银行卡号输入框,输入光大银行卡号(6226开头)
6. 页面显示光大银行协议,但**未勾选**协议
7. 直接点击"下一步"按钮
8. 观察结果

## 【期望结果】
1. 切换银行卡后,新银行的协议勾选状态应该重置为**未勾选**
2. 点击"下一步"时,应该弹出提示:"请阅读并勾选协议"
3. 必须手动勾选新银行的协议后才能提交

## 【实际结果】
1. 切换银行卡后,虽然显示了新银行协议,但协议勾选框状态未重置
2. 点击"下一步"时,没有校验新银行协议的勾选状态
3. 用户未勾选新银行协议也能成功提交

## 【附件】
- MongoDB日志: `icbc_request_log` 集合
- 测试卡号信息:

  {
    "建行卡": "6217 8570 0006 9016 192",
    "光大卡": "6226 6304 0794 9360",
    "extendInfo": "甚至工行支付签约V2"  // 注: 这里也有个错别字,应该是"深圳"
  }

## 【原因定位】
前端问题,可能原因:

1. **协议列表更新时未清空勾选状态**
   - 切换银行卡时,调用 `withhold/queryBankProtocol` 接口
   - 获取新银行协议后,动态追加到 `protocolList`
   - 但未清空或重置协议勾选状态

2. **勾选校验逻辑绑定错误**
   - 校验逻辑可能绑定在旧协议上
   - 切换银行卡后,旧协议勾选状态被误认为新协议已勾选

3. **协议唯一标识问题**
   - 协议勾选状态可能没有与银行卡号或协议URL绑定
   - 导致切换银行卡后仍使用旧的勾选状态

## 【修复建议】
前端修改建议:

1. **切换银行卡时重置协议状态**
   ```javascript
   // 在调用 withhold/queryBankProtocol 之前
   this.protocolList = [];  // 清空协议列表
   this.isAgreeProtocol = false;  // 重置勾选状态
   ```

2. **协议勾选状态与银行卡绑定**
   ```javascript
   // 使用 Map 存储不同银行卡的协议勾选状态
   this.protocolStatusMap = {
     [bankCardNo]: {
       protocolList: [],
       isAgree: false
     }
   };
   ```

3. **提交时增强校验逻辑**
   ```javascript
   // 检查当前显示的所有协议是否都已勾选
   if (this.protocolList.length > 0 && !this.isAgreeProtocol) {
     wx.showToast({
       title: '请阅读并勾选所有协议',
       icon: 'none'
     });
     return;
   }
   ```

4. **参考需求文档实现**
   - 按需求文档要求: "每次调用之前都清空之前银行协议数据"
   - 确保切换银行卡时,旧协议数据和勾选状态完全清除
