# BUG-010：预警计算逻辑错误导致误触发

## 【问题描述】
在2.0代扣绝对欠款预警功能中，预警触发的判断逻辑存在严重错误。无论当前值与历史平均值的实际差异如何，只要存在历史平均数据且参数配置大于0，系统就会触发预警，完全违背了"30%减量预警"的业务需求。

## 【测试场景】
**测试用例：** TC-005 预警阈值边界值测试

**测试配置：**
- 预警对象：2.0代扣绝对欠款 (early_warning_object = 2)
- 预警类型：减量预警 (early_warning_configure_type = 2)  
- 参数配置：30% (parameter_configure = 30)
- 车辆类型：客车 (car_type = 0)

## 【复现步骤】
1. 准备T-7天历史汇总数据：每天 `two_mode_absolute_arrears = 200,000`
2. 准备T日明细数据，使汇总后 `two_mode_absolute_arrears = 141,000`
3. 执行定时任务 `jobId=6` (arrearsCollectTask.getBlackArrearsDetailCount)
4. 检查预警记录生成情况

## 【日志记录&截图标注】

**历史数据验证：**
```sql
-- T-7天平均值计算
SELECT AVG(two_mode_absolute_arrears) as 'T-7平均值', COUNT(*) as '天数'
FROM fn_black_arrears_collect 
WHERE collect_time BETWEEN '2025-09-14' AND '2025-09-20' 
  AND car_type = 0;
-- 结果：平均值 = 200,000，天数 = 7
```

**当日汇总数据：**
```sql
-- T日汇总结果
SELECT collect_time, two_mode_absolute_arrears 
FROM fn_black_arrears_collect 
WHERE collect_time = '2025-09-21' AND car_type = 0;
-- 结果：2025-09-21, 141,000
```

**预警记录生成：**
```sql
-- 检查预警记录（错误地生成了）
SELECT id, early_warning_name, create_time
FROM fn_arrears_early_warning_record 
WHERE create_time > '2025-09-22 17:00:00'
ORDER BY create_time DESC;
-- 结果：生成了预警记录（错误行为）
```

**业务逻辑验证：**
```sql
SELECT 
  141000 as '当前T日值',
  200000 as 'T-7平均值',
  200000 * 0.7 as '预警阈值(70%)',
  CASE 
    WHEN 141000 < (200000 * 0.7) THEN '应该触发预警'
    ELSE '不应该触发预警'
  END as '预期结果';
-- 结果：不应该触发预警（141,000 > 140,000）
```

## 【核心代码问题】

**错误的计算逻辑：**
```java
// 位置：FnBlackArrearsCollectServiceImpl.addEarlyWarningRecord()
if (2 == earlyWarningConfigureType) {
    threshold = currentValue.add(BigDecimalUtils.multiply(averageValue, realRate, 2));
    if (currentValue.compareTo(threshold) < 0)
        addEarlyWarningRecord(configure, fnBlackArrearsCollectAdd.getCarType()); 
}
```

**实际计算过程：**
```java
// 代入实际值
currentValue = 141,000
averageValue = 200,000  
realRate = 0.3

threshold = currentValue + (averageValue × realRate)
         = 141,000 + (200,000 × 0.3)
         = 141,000 + 60,000
         = 201,000

// 判断条件
if (141,000 < 201,000) → TRUE → 触发预警
```

**逻辑简化分析：**
```java
// 简化后的判断条件
if (currentValue < currentValue + (averageValue × realRate))
// 等价于
if (0 < averageValue × realRate)
// 由于 averageValue > 0 且 realRate > 0，条件永远为 TRUE
```

## 【预期结果】
1. **正确的业务逻辑**：当前值相比平均值减少30%以上时才触发预警
2. **正确的计算公式**：
   ```java
   threshold = averageValue × (1 - realRate);  // 200,000 × 0.7 = 140,000
   if (currentValue < threshold) {             // 141,000 < 140,000 → FALSE
       addEarlyWarningRecord(configure, fnBlackArrearsCollectAdd.getCarType());
   }
   ```
3. **边界值测试**：141,000 > 140,000（阈值）→ 不应触发预警
4. **数据一致性**：不应该在 `fn_arrears_early_warning_record` 表中创建记录

**具体期望：**
- 当前值 141,000 > 预警阈值 140,000 → 不触发预警
- `fn_arrears_early_warning_record` 表中不应该有新增记录
- 边界值测试能够正确区分触发和未触发的情况

---

**技术分析：**
- **根本原因**：预警计算公式设计错误，将减量判断写成了增量判断
- **影响范围**：所有使用 `early_warning_configure_type = 2` 的减量预警配置
- **严重程度**：高 - 核心业务逻辑错误，导致预警功能完全失效
- **涉及代码**：`FnBlackArrearsCollectServiceImpl.addEarlyWarningRecord()` 方法
- **修复建议**：重写预警判断逻辑，将公式修正为正确的减量计算方式
