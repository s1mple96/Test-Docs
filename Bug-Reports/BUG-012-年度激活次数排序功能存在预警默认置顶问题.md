# BUG-012：年度二次激活次数排序功能存在"预警默认置顶"旧功能干扰

## 【问题描述】
在三个系统（任货行、任通行、新发行）的年度二次激活次数排序功能中，存在"预警默认置顶"的旧功能逻辑，该功能会干扰正常的排序逻辑，导致用户按年度激活次数进行升序或降序排序时，结果不符合预期。

## 【涉及系统】
- ✅ **任货行(fenmi_etc)**：年度激活次数排序功能
- ✅ **任通行(RTX)**：年度激活次数排序功能  
- ✅ **新发行**：年度激活次数排序功能

## 【复现步骤】
1. 登录任意一个系统的管理后台
2. 进入二次激活记录管理页面
3. 查询包含不同年度激活次数的数据
4. 点击"年度二次激活次数"字段进行排序
5. 选择升序排序（由低到高）
6. 观察排序结果

## 【实际结果】
**排序异常现象：**
- 某些记录（疑似预警数据）始终显示在列表顶部
- 正常的升序/降序排序逻辑被打乱
- 年度激活次数为0或较少的记录可能不在最前面（升序时）
- 年度激活次数最多的记录可能不在最前面（降序时）

**示例异常排序结果（降序）：**
```
排序结果：
[预警记录] 年度激活次数: 5  ← 置顶显示
[预警记录] 年度激活次数: 3  ← 置顶显示
[正常记录] 年度激活次数: 25 ← 应该排在最前面
[正常记录] 年度激活次数: 20
[正常记录] 年度激活次数: 15
```

## 【预期结果】
**正确的排序逻辑：**

**降序排序结果（由高到低）：**
```
年度激活次数: 25
年度激活次数: 20  
年度激活次数: 15
年度激活次数: 5
年度激活次数: 3
年度激活次数: 0
```

**升序排序结果（由低到高）：**
```
年度激活次数: 0
年度激活次数: 3
年度激活次数: 5
年度激活次数: 15
年度激活次数: 20
年度激活次数: 25
```

## 【问题分析】
**可能原因：**
1. **旧功能残留**：系统中存在预警数据默认置顶的排序逻辑
2. **排序优先级冲突**：预警状态的排序优先级高于年度激活次数排序
3. **SQL排序逻辑复杂**：ORDER BY语句中可能包含多个排序条件

**推测的问题SQL：**
```sql
-- 可能存在的问题排序逻辑
SELECT * FROM activate_records 
ORDER BY 
    is_warning DESC,  -- 预警优先置顶
    annual_activate_count DESC  -- 年度激活次数排序
```

**应该修正为：**
```sql
-- 正确的排序逻辑
SELECT * FROM activate_records 
ORDER BY annual_activate_count DESC  -- 仅按年度激活次数排序
```

## 【修复建议】
1. **移除预警置顶逻辑**：在年度激活次数排序功能中，去除所有预警相关的排序优先级
2. **简化排序条件**：确保排序只按用户选择的年度激活次数进行（升序/降序）
3. **三系统统一**：确保任货行、任通行、新发行三个系统的排序逻辑完全一致
4. **保留预警标识**：可以保留预警的视觉标识（如颜色、图标），但不影响排序逻辑

## 【验证方案】
**测试数据准备：**
```
测试数据集：
车辆A - 年度激活次数: 0,  预警状态: 是
车辆B - 年度激活次数: 25, 预警状态: 否  
车辆C - 年度激活次数: 5,  预警状态: 是
车辆D - 年度激活次数: 15, 预警状态: 否
车辆E - 年度激活次数: 3,  预警状态: 是
```

**验证步骤：**
1. **降序排序验证**：期望顺序 B(25) → D(15) → C(5) → E(3) → A(0)
2. **升序排序验证**：期望顺序 A(0) → E(3) → C(5) → D(15) → B(25)
3. **三系统一致性**：在三个系统中使用相同数据验证排序结果一致

## 【影响评估】
- **用户体验**：排序功能不可用，影响数据查看和分析
- **数据分析**：无法按激活次数进行有效的数据排序分析
- **功能完整性**：影响新增排序功能的核心价值
- **系统一致性**：三个系统的排序逻辑不统一
