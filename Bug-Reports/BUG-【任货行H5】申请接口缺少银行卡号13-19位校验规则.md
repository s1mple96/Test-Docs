# BUG-【任货行H5】银行卡号校验失效-checkType拼写错误导致

## 【BUG描述】
任货行H5申请页面的银行卡号校验规则因 `checkType` 字段拼写错误(`sting` 写成了 `string`),导致13-19位的校验规则完全失效,不符合规范的银行卡号(如12位、20位、包含字母等)可以直接通过前端校验并提交成功,存在严重的数据合规性风险。影响范围：任货行H5所有申请场景。

## 【前置条件】
- **测试环境**: 192.168.1.60 (测试环境)
- **测试账号**: 任意用户账号
- **访问方式**: H5页面
- **相关接口**: `com.hcb.submitApplyBankInfo`
- **相关页面**: 任货行H5银行卡信息提交页面

## 【复现步骤】
1. 打开任货行H5申请页面
2. 进入银行卡信息填写页面
3. 上传身份证正反面、填写个人信息
4. 在银行卡号输入框输入 **12位数字**（如：`123456789012`）
5. 填写其他必填信息(银行预留手机号等)
6. 点击"提交"按钮
7. 观察是否允许提交

**测试数据:**
- 12位卡号: `123456789012` (应拒绝,实际可能通过)
- 20位卡号: `12345678901234567890` (应拒绝,实际可能通过)
- 16位含字母: `622202123456789A` (应拒绝,实际可能通过)

## 【期望结果】
根据测试用例 `【任货行小程序+H5】申请接口增加银行卡位数校验测试用例_V1.0.md` 的要求:

1. **前端校验**:
   - 银行卡号长度必须在 **13-19位** 之间
   - 必须是 **纯数字**
   - 不符合规则时,前端拦截提交并提示: "请输入13-19位有效银行卡号"

2. **后端校验**:
   - 接口 `com.hcb.submitApplyBankInfo` 必须校验银行卡号长度 **13-19位**
   - 必须校验是否为 **纯数字格式**
   - 不符合规则时,返回失败响应: "请输入13-19位有效银行卡号"

3. **校验规则参考** (小程序已有):
```javascript
{
    name: "bankNo",
    checkType: "string",
    checkRule: "13,19",
    errorMsg: "银行卡号不合法,请确认"
}
```

## 【实际结果】

### 1. H5前端校验情况
**校验规则失效 - checkType拼写错误** 🔴

**问题代码**:
```javascript
{
    name: "bankNo",
    checkType: "sting",  // ❌ 拼写错误! 应该是 "string"
    checkRule: "13,19",
    errorMsg: "银行卡号不合法,请确认"
}
```

**根本原因**:
- H5前端有配置13-19位的校验规则
- 但是 `checkType` 字段拼写错误: **`sting`** 而不是 **`string`**
- 导致校验框架无法识别该规则类型
- 整个校验规则被忽略,相当于没有任何校验
- 所有不合法的银行卡号都能通过前端验证

### 2. 后端接口校验情况  
**校验不完整** ⚠️

代码位置: `E:\后端代码\hcbapi\hcbapi\src\main\java\com\hcb\api\service\truck\trucketcapply\impl\ApplyOrderCommonService.java`

**当前代码** (第1552-1557行):
```java
if (StringUtils.isEmpty(bankNo)) {

if(bankNo.length() > 20){
```

**问题分析**:

- ❌ **缺少最小长度13位的校验**
- ❌ **缺少纯数字格式校验**

## 【原因定位】

### 1. H5前端原因
**根本原因**: `checkType` 字段拼写错误

**错误代码**:
```javascript
checkType: "sting"  // ❌ 错误拼写
```

**正确代码**:
```javascript
checkType: "string"  // ✅ 正确拼写
```

**影响分析**:
1. 前端校验框架无法识别 `"sting"` 这个类型
2. 导致整个 `checkRule: "13,19"` 规则被忽略
3. 银行卡号输入框失去了长度校验保护
4. 任何长度的输入都能通过前端验证
5. 包括字母、特殊字符等也无法被拦截

### 2. 后端接口原因
**根本原因**: 代码实现不完整

**代码位置**: 
- `ApplyOrderCommonService.java` 第1555-1557行
- `HcbSelfHelpApplyService.java` 第854-856行  
- `ChannelApplyOrderService.java` 第648-650行

**问题分析**:
1. 早期代码只做了基础的非空和最大长度校验
2. 后续需求新增了13-19位的要求,但后端代码未及时更新
3. 小程序前端已更新规则,但后端和H5前端遗漏
4. 缺少纯数字格式的正则校验

### 3. 前后端不一致
- 小程序前端: `checkType: "string"` ✅ 拼写正确,校验生效
- H5前端: `checkType: "sting"` ❌ 拼写错误,校验失效  
- 后端三个服务类都未更新完整校验 ❌

### 4. 拼写错误的严重性
这是一个 **低级但致命** 的拼写错误:
- ✅ 有配置规则(`checkRule: "13,19"`)
- ✅ 有错误提示(`errorMsg`)
- ❌ 但因为 `checkType` 拼写错误,整个规则不生效
- 🔴 开发和测试都没有发现这个问题
- 🔴 导致不合规数据进入系统

**或者使用自定义校验函数**:
```javascript
// 银行卡号校验
validateBankNo(value) {
    if (!value) {
        return '请输入银行卡号';
```
