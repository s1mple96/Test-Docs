## 关键字段需求映射

| 需求原文描述 | RTX（rtx库） | 新发行（fenmi_etc库） |
|-------------|------------|---------------------|
| 扣款状态=扣款失败 | `rtx_etc_deducrecord_txb.status = '2'`（代扣失败） | `etc_deduct_record.status = 2` |
| 创建时间=近3天（含当天） | `rtx_etc_deducrecord_txb.create_time >= now()-3天` | `etc_deduct_record.create_time >= now()-3天`（72小时滑动窗口，非自然日，需与产品确认语义） |
| 最近一次扣款时间 | `rtx_etc_deducrecord_txb.deduc_time`（注意：无t，是`deduc_time`） | `etc_deduct_record.deduct_time`（有t，是`deduct_time`） |
| 排除20分钟内有扣款动作 | `deduc_time <= now()-20分钟` | `deduct_time <= now()-20分钟` |
| 钱包1.0用户 | `rtx_user_wallet.wallet_mode = '1'` | 扣款顺序配置 `etc_wallet_deduction_method.deduction_method IN (1,2)` |
| 钱包2.0代扣用户 | `rtx_user_wallet.wallet_mode = '2'` | 扣款顺序配置 `etc_wallet_deduction_method.deduction_method = 3`（微信代扣） |
| 新发行所有用户 | 不涉及（RTX无"新发行"概念） | `etc_deduct_record`中所有运营商（TXB/MTK） |
| 任务执行时间控制（6-18点/2小时） | **代码内置**：`hour < 6 \|\| hour > 18` + Redis锁（TTL=2小时）`FAIL_WITHHOLD_RUN_TASK_CACHE_KEY` | **外部Quartz cron**配置，代码无内置时间判断 |
| 补扣不触发短信 | `BalanceBillDeductProcessor`：`FAIL_SUPPLEMENTARY_DEDUCT`场景传`false`，不调用`asyncBalanceDeductComplete` | 短信仅在`FIRST_DEDUCT`触发，补扣走`FAIL_SUPPLEMENTARY_DEDUCT`枚举，天然不触发 |
| 垫付补扣任务改动（任通行特有） | `walletAdvanceDeductScene`新增`deduc_time <= now()-20分钟`过滤 | 无此概念 |

## 一、任通行（RTX）— 新增补扣任务

### 1.1 基本功能

#### TC-001 补扣任务正常触发：扣款失败+近3天+最后扣款超过20分钟的账单被处理

- **前置条件**:
  - 系统：任通行定时任务（rtx-quartz）
  - 数据库：rtx
  - 当前时间在 06:00–18:00（代码内置判断）
  - 造数 SQL（确认或插入满足条件的账单）：
    ```sql
    SELECT id, status, deduc_time, create_time, operator_code
    FROM rtx.rtx_etc_deducrecord_txb
    WHERE status = '2'
      AND create_time >= DATE_SUB(NOW(), INTERVAL 3 DAY)
      AND deduc_time <= DATE_SUB(NOW(), INTERVAL 20 MINUTE)
    ORDER BY id DESC LIMIT 5;
    ```
  - 若无，手动插入一条 TXB 账单（status='2'，create_time=当前时间，deduc_time=当前时间-30分钟）

- **测试步骤**:
  1. 手动触发 `BillDeductPayTask.failSupplementaryDeductTask`
  2. 观察日志：确认打印"补扣失败账单任务 start"和"补扣失败账单任务 结束"
  3. 查询账单状态变化：
     ```sql
     SELECT id, status, deduc_time, update_time
     FROM rtx.rtx_etc_deducrecord_txb
     WHERE id = {测试账单ID} LIMIT 1;
     ```

- **预期结果**:
  - 账单被捞取，发起 `FAIL_SUPPLEMENTARY_DEDUCT` 扣款
  - `deduc_time` 更新为当前时间；status 根据结果变更（成功→'1'，失败→'2'/'4'/'5'，支付中→'3'）

---

#### TC-002 任务执行时间控制：6-18点内代码放行，非窗口时段自动跳过

- **前置条件**:
  - 数据库：rtx
  - 存在满足条件的扣款失败账单

- **测试步骤**:
  1. **时间窗口内**（如10:00）：触发 `failSupplementaryDeductTask`，查看日志
  2. **时间窗口外**（如23:00，需修改系统时间或等待观察）：触发任务，查看日志

- **预期结果**:
  - 窗口内：任务正常执行，打印"补扣失败账单任务 start"
  - 窗口外：日志打印"补扣失败账单任务没在指定时间内（6-18点且间隔2h）或未执行完，跳过"，**不处理任何账单**

---

#### TC-003 Redis锁防止2小时内重复执行

- **前置条件**:
  - 数据库：rtx；Redis：192.168.1.60（pwd: chefu666）
  - 任务正在执行中（或手动写入Redis锁 key = `FAIL_WITHHOLD_RUN_TASK_CACHE_KEY`）

- **测试步骤**:
  1. 手动在 Redis 中写入锁 key：`SET FAIL_WITHHOLD_RUN_TASK_CACHE_KEY 1 EX 7200`
  2. 再次触发 `failSupplementaryDeductTask`

- **预期结果**:
  - 第二次触发被跳过，日志打印"跳过"信息
  - 确保任务不会并发重复执行（需求：「执行时间和间隔2小时」）
