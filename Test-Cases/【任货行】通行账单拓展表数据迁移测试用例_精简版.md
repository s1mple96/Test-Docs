## 4. 幂等性和并发测试（重点）⭐⭐⭐

### 4.1 MongoDB唯一索引验证（基础设施检查）

#### TC-011 MongoDB唯一索引存在性验证⭐⭐⭐
- **前置条件**：无
- **测试步骤**：
  ```javascript
  // 在MongoDB中执行
  db.hcb_deduct_record_extend_backup.getIndexes()
  ```
  检查是否存在id字段的唯一索引
- **预期结果**：
  - ✅ **必须存在**：`{ "id": 1 }, unique: true`
  - ❌ 如果不存在：**高风险**，并发时可能重复插入
- **优先级**：P0
- **修复建议**（如果不存在）：
  ```javascript
  db.hcb_deduct_record_extend_backup.createIndex(
    { "id": 1 },
  ```

### 4.2 exists()查询机制验证

#### TC-012 exists()查询生效验证⭐⭐⭐
- **前置条件**：MongoDB中已存在ID=999999的测试数据
- **测试步骤**：
  1. MySQL中插入ID=999999的数据（符合迁移条件）
  2. 执行迁移
  3. 观察日志，检查是否跳过该ID
  4. 检查MongoDB中ID=999999的数据数量（应该仍为1）
- **预期结果**：迁移任务跳过已存在的ID，不重复插入
- **优先级**：P0

#### TC-013 重复执行幂等性验证⭐⭐⭐
- **前置条件**：MySQL中存在10条需迁移数据（ID: 1-10）
- **测试步骤**：
  1. 第1次执行迁移任务，记录结果：MySQL删除10条，MongoDB插入10条
  2. MySQL中手动重新插入相同的10条数据（相同ID）
  3. 再次执行迁移任务
  4. 检查MongoDB中数据量（应该仍为10条，不是20条）
- **预期结果**：exists()查询生效，不重复插入，MySQL数据被删除
- **优先级**：P0

### 4.3 并发执行测试（核心风险）

#### TC-014 并发执行数据竞争测试⭐⭐⭐
- **前置条件**：MySQL中存在100条需迁移数据，MongoDB中不存在这些数据
- **测试步骤**：
  1. 同时启动2个迁移任务（两个HTTP请求并发调用）
  2. 等待两个任务都完成
  3. 检查MongoDB中数据量
  4. 检查是否有重复ID的数据
- **预期结果**：
  - **如果有唯一索引**：MongoDB中100条数据，第二个任务插入时报错（可接受）
  - **如果无唯一索引**：❌ **高风险**，MongoDB中可能有重复数据
- **优先级**：P0

#### TC-015 并发时exists()查询竞态条件测试⭐⭐⭐
- **前置条件**：MySQL中存在50条需迁移数据
- **测试步骤**：
  ```
  时间线模拟：
  T1: 任务A执行 exists(ID=1) → false
  T2: 任务B执行 exists(ID=1) → false (A还没插入)
  T3: 任务A执行 insert(ID=1) → 成功
  T4: 任务B执行 insert(ID=1) → ？
  ```
  1. 并发启动2个迁移任务
  2. 使用较小的批次大小增加竞争概率
  3. 检查最终数据
- **预期结果**：
  - **有唯一索引**：任务B插入时报错，数据正确（100条）✅
  - **无唯一索引**：数据重复（可能>100条）❌
- **优先级**：P0

#### TC-016 高并发压测（1000并发）⭐⭐

## 5. 异常处理测试

#### TC-019 MongoDB写入超时但实际成功测试⭐⭐⭐
- **前置条件**：模拟MongoDB慢查询或网络延迟
- **测试步骤**：
  1. 调整MongoDB连接超时为1秒（很短）
  2. 插入大数据量或在MongoDB上执行慢查询
  3. 执行迁移，可能触发超时异常
  4. 检查MongoDB中是否实际写入了数据
  5. 检查MySQL数据状态
- **预期结果**：
  - **最佳情况**：异常被捕获，MySQL未删除，下次重跑时exists()跳过 ✅
  - **风险情况**：MySQL被删除但无数据写入MongoDB ❌
- **优先级**：P0
- **验证点**：异常处理逻辑是否完善

#### TC-020 exists()查询超时测试⭐⭐
- **前置条件**：模拟MongoDB查询延迟

  1. 在MongoDB上执行大量慢查询占用资源
  2. 执行迁移任务
  3. 观察exists()查询是否超时
- **预期结果**：
  - 超时后有重试机制，或任务失败但数据安全

## 给开发的建议

### 🔴 当前方案仍存在的风险

#### 1. 并发竞态条件（时间窗口问题）
```java
// 问题代码示例
if (!mongoTemplate.exists(id)) {  // T1: 线程A查询 → false
                                   // T2: 线程B查询 → false (A还没插入)
    mongoTemplate.insert(data);    // T3: 线程A插入成功
                                   // T4: 线程B也插入成功 ❌（如果无唯一索引）
    mysqlMapper.delete(id);
}
```

**解决方案**：
- ✅ **必须添加MongoDB唯一索引**（最简单有效）
- 或者添加分布式锁（复杂）

#### 2. MongoDB异常但实际成功
```java
try {
    if (!exists(id)) {
        insert(data);  // 超时异常，但实际已写入
        delete(id);    // 不执行
    }
} catch (TimeoutException e) {
    log.error("写入失败", e);
    // MySQL数据未删除，下次exists=true会跳过 ✅
    // 但需要确保异常处理逻辑正确
}
```

**建议**：
- 添加完善的异常处理
- 超时后再次查询确认是否真的写入

#### 3. 性能问题（10万次exists查询）
```java
// 当前逐条查询
for (Data data : dataList) {
    if (mongoTemplate.exists(data.getId())) continue;
    // ...
}

// 优化：批量查询
Set<Long> existingIds = batchQueryExistingIds(dataList);
for (Data data : dataList) {
    if (existingIds.contains(data.getId())) continue;
    // ...
}
```
