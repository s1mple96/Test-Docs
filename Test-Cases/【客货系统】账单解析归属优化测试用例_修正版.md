# 客货系统相同运营商账单解析归属优化测试用例（修正版）

## 1. ETC卡号库存查询优先逻辑测试

### 1.1 ETC卡号查询基础功能测试

#### TC-001 ETC卡号用户表查询成功测试
- **前置条件**：广西JTK或华软三省（MTK/LTK/XTK）运营商推送账单文件，账单包含ETC卡号，用户系统（hcb_truckuser表）中存在对应ETC卡号记录
- **测试步骤**：
  1. 准备测试账单数据，包含有效的ETC卡号（如：15012315231921226552），运营商代码为JTK或MTK
  2. 确保用户系统（hcb_truckuser表）中存在该ETC卡号（ETC_SN字段）对应的用户记录
  3. 触发账单解析入库流程
  4. 验证系统首先通过ETC卡号查询用户系统（hcb_truckuser表）
  5. 检查第一层查询结果和归属系统的确认
  6. 验证账单最终归属到正确的系统
- **预期结果**：系统优先通过ETC卡号在用户表查询成功，准确获取用户归属信息，账单正确归属到对应系统

#### TC-002 ETC卡号库存查询成功测试
- **前置条件**：广西JTK或华软三省（MTK/LTK/XTK）运营商推送账单文件，账单包含ETC卡号，库存系统（hcb_newstock表）中存在对应ETC卡号记录
- **测试步骤**：
  1. 准备测试账单数据，包含有效的ETC卡号（如：15012315231921226552），运营商代码为JTK或MTK
  2. 确保库存系统（hcb_newstock表）中存在该ETC卡号（EXTERNAL_DEVICE_NO字段）对应的车辆和系统归属信息
  3. 触发账单解析入库流程
  4. 验证系统优先通过ETC卡号查询库存系统（hcb_newstock表）
  5. 检查查询结果和归属系统的确认
  6. 验证账单最终归属到正确的系统
- **预期结果**：系统优先通过ETC卡号查询库存，准确获取车辆归属信息，账单正确归属到对应系统

#### TC-003 ETC卡号库存查询失败测试
- **前置条件**：广西JTK或华软三省运营商推送账单文件，账单包含ETC卡号，但库存系统（hcb_newstock表）中不存在该ETC卡号记录
- **测试步骤**：
  1. 准备测试账单数据，包含库存系统中不存在的ETC卡号，运营商代码为JTK或MTK
  2. 触发账单解析入库流程
  3. 验证ETC卡号在库存系统（hcb_newstock表）查询返回空结果
  4. 确认系统自动切换到原有逻辑（车牌号+颜色查询用户表hcb_truckuser）
  5. 检查兜底逻辑的执行结果
  6. 验证最终归属结果的正确性
- **预期结果**：ETC卡号查询失败后，系统自动执行兜底逻辑，通过车牌号和颜色确认归属

#### TC-004 ETC卡号为空或无效测试
- **前置条件**：广西JTK或华软三省运营商推送账单文件，ETC卡号字段为空、null或格式无效
- **测试步骤**：
  1. 准备测试账单数据，ETC卡号字段为空值，运营商代码为JTK或MTK
  2. 准备测试账单数据，ETC卡号字段为null
  3. 准备测试账单数据，ETC卡号格式无效
  4. 分别触发账单解析流程
  5. 验证系统对无效ETC卡号的处理逻辑
  6. 确认直接执行兜底逻辑
- **预期结果**：ETC卡号无效时，系统跳过ETC查询，直接执行车牌号+颜色的原有逻辑

### 1.2 ETC卡号与车牌号匹配验证测试

#### TC-005 ETC卡号与车牌号一致性验证测试
- **前置条件**：库存系统（hcb_newstock表）中ETC卡号对应的车牌号与账单中的车牌号需要保持一致
- **测试步骤**：
  1. 准备账单数据，包含ETC卡号和车牌号信息，运营商代码为JTK或MTK
  2. 确保库存系统（hcb_newstock表）中该ETC卡号（EXTERNAL_DEVICE_NO）对应的车牌号（CAR_NUM）与账单一致
  3. 执行账单解析流程
  4. 验证ETC卡号查询结果的车牌号匹配检查
  5. 确认一致性验证通过后的归属结果
- **预期结果**：ETC卡号查询成功且车牌号匹配一致，账单正确归属

#### TC-006 ETC卡号与车牌号不一致处理测试
- **前置条件**：库存系统（hcb_newstock表）中ETC卡号对应的车牌号与账单中的车牌号不一致
- **测试步骤**：
  1. 准备账单数据，ETC卡号与车牌号存在不匹配情况，运营商代码为JTK或MTK
  2. 执行账单解析流程
  3. 验证系统对不一致情况的处理逻辑
  4. 确认是否执行额外的验证或容错处理
  5. 检查最终的归属决策逻辑
- **预期结果**：系统能合理处理ETC卡号与车牌号不一致的情况，有明确的处理策略

#### TC-007 多个ETC卡号对应同一车牌测试
- **前置条件**：库存系统（hcb_newstock表）中可能存在多个ETC卡号对应同一车牌的情况
- **测试步骤**：
  1. 在库存系统（hcb_newstock表）中设置多个ETC卡号对应同一车牌
  2. 使用其中一个ETC卡号的账单进行测试，运营商代码为JTK或MTK
  3. 验证系统查询逻辑的准确性
  4. 检查是否正确识别有效的ETC卡号
  5. 确认归属结果的唯一性和正确性
- **预期结果**：系统能正确处理一对多的ETC卡号关系，准确确定归属

## 2. 车牌号车牌颜色兜底逻辑测试

### 2.1 原有逻辑回归测试

#### TC-008 车牌号颜色查询原逻辑回归测试
- **前置条件**：ETC卡号查询失败或无效，需要执行原有的车牌号+颜色查询逻辑（查询hcb_truckuser表）

  1. 准备ETC卡号无效的账单数据，运营商代码为JTK或MTK
  2. 确保账单包含有效的车牌号和颜色信息
  3. 执行账单解析流程

  6. 对比修改前后的处理结果一致性
- **预期结果**：原有车牌号+颜色查询逻辑正常工作，处理结果与修改前保持一致

#### TC-009 车牌号颜色匹配唯一用户测试
- **前置条件**：用户系统（hcb_truckuser表）中存在唯一的车牌号+颜色匹配用户
- **测试步骤**：
  1. 确保测试车牌号+颜色在用户系统（hcb_truckuser表）中只有一个匹配用户
  2. 准备对应的账单数据，运营商代码为JTK或MTK
  3. 执行兜底逻辑查询
  4. 验证用户匹配的准确性
  5. 确认账单归属的正确性
- **预期结果**：唯一匹配用户的情况下，账单准确归属到对应系统

#### TC-010 车牌号颜色匹配多用户状态优先级测试
- **前置条件**：用户系统（hcb_truckuser表）中存在相同车牌号+颜色的多个用户，需要按用户状态优先级选择
- **测试步骤**：
  1. 设置相同车牌号+颜色但不同状态的多个用户在hcb_truckuser表中
  2. 用户状态包括：正常(STATUS='1')、暂无ETC状态(STATUS='0')、ETC黑名单(STATUS='2')、待激活(STATUS='3')、已注销(STATUS='4')
  3. 准备测试账单数据，运营商代码为JTK或MTK
  4. 执行兜底逻辑查询
  5. 验证系统按优先级选择：正常 > 暂无ETC状态 > ETC黑名单 > 待激活 > 已注销
  6. 确认选择优先级最高的用户进行账单归属
- **预期结果**：系统按照用户状态优先级规则选择用户，确保账单归属的准确性和一致性

#### TC-013 查询服务异常容错测试
- **前置条件**：库存系统查询服务可能出现异常或超时
- **测试步骤**：
