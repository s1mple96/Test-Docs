# 【任货行】广州工行直接支付转账接入测试用例

> **需求范围**: 任货行小程序-广州工行直接支付转账接入  
> **涉及系统**: 任货行小程序（hcbapi）  
> **数据库**: hcb  
> **测试环境**: 192.168.1.60  
> **银行通道**: GZ_ICBC（广州工行工银e企付）  
> **接口说明**: 会员档案录入(memberapply/V1)、会员档案查询(memberquery/V1)、支付申请(cppayapply/V2)、支付信息查询(cporderquery/V2)、支付确认(cppreservationpay/V2)

---

## 1. 申办流程-开户测试

### 1.1 新用户开户流程

#### TC-001 新用户首次申办-走广州工行开户
- **前置条件**:
  - 系统: 任货行小程序（申办流程）
  - 数据库: hcb
  - 场景: 身份证号对应用户在 `hcb_bank_account` 中无 `bank_code='GZ_ICBC'` 的记录
  - 确认该用户钱包已存在（`hcb_truckuserwallet` 中有记录）
  - 用户已完成实名认证
  - 测试环境: 192.168.1.60
- **测试步骤**:
  1. 打开任货行小程序，以新用户身份进入申办流程
  2. 填写申办信息（姓名、身份证、绑定银行卡号、开户行）
  3. 提交申办
  4. 查询数据库：`SELECT id, member_no, account_status, reason FROM hcb_bank_account WHERE id_code='xxx' AND bank_code='GZ_ICBC'`
- **预期结果**:
  - 系统调用广州工行 `memberapply/V1` 接口发起会员档案录入
  - `hcb_bank_account` 新增一条 `bank_code='GZ_ICBC'` 的记录
  - `account_status=4`（开户中，待线下审核）或 `account_status=1`（已成功）
  - `member_no` 字段有值（格式 M+6位序号，如 M000001）
  - MongoDB `gz_icbc_request_log` 中有"会员档案录入"的请求日志，`returnCode=0`

#### TC-002 新用户开户-工行账户为本行（工商银行）
- **前置条件**:
  - 同 TC-001
  - 绑定银行卡为工商银行卡（`accBankFlag=1`）
- **测试步骤**:
  1. 以工商银行卡用户申办
  2. 提交申办，触发开户
  3. 查询 MongoDB 日志中请求参数
- **预期结果**:
  - 请求参数中 `accBankFlag=1`，`accBankNo` 和 `accBankName` 为空
  - 正常完成开户流程，无报错

#### TC-003 新用户开户-工行账户为他行（非工商银行）
- **前置条件**:
  - 同 TC-001
  - 绑定银行卡为他行（如建设银行、招商银行等，`accBankFlag=2`）
- **测试步骤**:
  1. 以他行银行卡用户申办
  2. 提交申办，触发开户
  3. 查询 MongoDB 日志中请求参数
- **预期结果**:
  - 请求参数中 `accBankFlag=2`，`accBankNo`（联行号）和 `accBankName`（开户行名称）必须有值
  - 不能发起 `accBankNo` 或 `accBankName` 为空的请求

#### TC-004 新用户开户成功后-再次申办不重复开户
- **前置条件**:
  - 系统: 任货行小程序
  - 数据库: hcb
  - 该用户 `hcb_bank_account` 中已有 `bank_code='GZ_ICBC'` 且 `account_status=1`（已成功）的记录
  - 该记录 `member_no` 字段有值
- **测试步骤**:
  1. 以已开户成功的用户再次进入申办流程
  2. 触发开户查询接口
  3. 观察是否重新调用开户接口
- **预期结果**:
  - 系统不再重新调用广州工行 `memberapply` 接口
  - 日志输出提示"已经提交开户或已开户成功，无需重复开户"
  - `hcb_bank_account` 记录数不增加

#### TC-005 新用户开户中状态（account_status=4）-再次申办不重复开户
- **前置条件**:
  - 数据库中已存在 `bank_code='GZ_ICBC'` 且 `account_status=4`（开户中）的记录
- **测试步骤**:
  1. 以开户中状态的用户再次进入申办流程
  2. 触发开户查询接口
- **预期结果**:
  - 系统不再重新调用广州工行 `memberapply` 接口
  - 提示信息正确，不重复开户

#### TC-006 开户失败（account_status=5）-重新开户
- **前置条件**:
  - 数据库中存在 `bank_code='GZ_ICBC'` 且 `account_status=5`（失败）的记录
  - 例如：`SELECT id, member_no, account_status, reason FROM hcb_bank_account WHERE bank_code='GZ_ICBC' AND account_status=5 LIMIT 1`
- **测试步骤**:
  1. 以开户失败状态的用户重新提交申办
  2. 系统触发重新开户流程
  3. 查询数据库和 MongoDB 日志
- **预期结果**:
  - 系统重新调用广州工行 `memberapply` 接口
  - 重新开户时使用**原会员编码**（`memberNo` 保持不变，不重新生成）
  - 开户状态更新，备注包含"重新开户"相关信息

## 3. 余额刷新测试

### 3.1 余额刷新正常流程

#### TC-013 余额刷新-广州工行转账流水查询
- **前置条件**:
  - 系统: 任货行小程序（余额刷新）
  - 数据库: hcb
  - 用户 `hcb_bank_account` 中 `bank_code='GZ_ICBC'`，`account_status=1`，`member_no` 有值
  - 广州工行账户存在历史转账记录
- **测试步骤**:
  1. 在任货行小程序点击"余额刷新"
  2. 系统调用广州工行 `tradefundlistquery/V1`（按会员进行明细查询）
  3. 查询 MongoDB 日志
- **预期结果**:
  - 接口调用成功，`returnCode=0`
  - 返回该会员的转账流水列表
  - 日志中有"按会员查询转账流水"的请求记录

#### TC-014 余额刷新-根据转账流水发起支付申请
- **前置条件**:
  - 用户广州工行账户存在待处理的转账记录
  - `hcb_bank_transferrecord` 中该 `member_no` 有新增的转账记录（status 为待支付状态）
- **测试步骤**:
  1. 触发余额刷新
  2. 系统获取到转账流水后，调用 `cppayapply/V2` 发起支付申请
  3. 查询 MongoDB 日志中支付申请请求
- **预期结果**:
  - 调用广州工行 `cppayapply/V2`，支付方式为担保支付（`payMode=3`，`operType=301`）
  - 请求中 `orderAmount`、`sumPayamt` 单位为分（正确转换）
  - 响应 `returnCode=0`，支付申请成功
  - `hcb_bank_transferrecord` 更新转账记录的申请流水号

#### TC-015 余额刷新-支付申请成功后查询支付状态
- **前置条件**:
  - `hcb_bank_transferrecord` 已有支付申请流水号，但支付状态为进行中（`status=3`）
- **测试步骤**:
  1. 触发余额刷新，系统检测到支付申请状态为进行中（payStatus=3）
  2. 系统自动重试查询（最多3次，间隔1秒）
  3. 查看日志是否有"广州工行-支付申请查询 支付中状态--再次申请查询开始"
- **预期结果**:
  - 系统重试查询，日志显示重试记录
  - 最终如果状态仍为进行中，抛出异常并记录日志
  - 不会无限重试（超过3次后停止）

#### TC-016 余额刷新-支付申请查询成功后发起确认支付
- **前置条件**:
  - 支付申请成功，`hcb_bank_transferrecord` 中记录有 `transfer_applay_flow`（支付申请流水号）
- **测试步骤**:
  1. 触发余额刷新
  2. 支付申请查询成功（非进行中状态）
  3. 系统调用 `cppreservationpay/V2` 发起确认支付（`operType=302`）
  4. 查询 MongoDB 日志和数据库
- **预期结果**:
  - 确认支付请求发起成功，`returnCode=0`
  - `partnerSeqOrigin` 与支付申请的 `partnerSeq` 一致（原流水号关联）
  - `hcb_bank_transferrecord` 更新转账流水号（`transfer_flow`）和状态为成功（`status=4`）
  - 钱包余额增加对应充值金额

#### TC-017 余额刷新-确认支付成功-钱包余额更新
- **前置条件**:
  - `hcb_bank_transferrecord` 中有 `bank_code='GZ_ICBC'` 且 `status=4`（支付成功）的记录
  - 对应充值记录 `hcb_topup_record` 中状态应为成功
- **测试步骤**:
  1. 余额刷新整个流程完成后，查询钱包余额
  2. `SELECT * FROM hcb_truckuserwallet WHERE truck_user_id='xxx'`
  3. 查询充值记录 `SELECT * FROM hcb_topup_record WHERE corp_serno='xxx'`
- **预期结果**:
  - 钱包余额增加了对应的充值金额
  - `hcb_topup_record` 有对应的充值成功记录
  - 充值金额正确（转换前单位分，充值入账单位元，精度正确）

## 6. 数据库验证SQL参考

```sql
-- 查询广州工行开户记录
SELECT id, member_no, account_status, account_bank_name, phone_number, reason, create_time
FROM hcb_bank_account
WHERE bank_code = 'GZ_ICBC'
ORDER BY create_time DESC LIMIT 20;

-- 查询广州工行转账记录
SELECT id, member_no, status, amount, remark, charge_order_no, 
       transfer_applay_flow, transfer_flow, create_time
FROM hcb_bank_transferrecord
WHERE bank_code = 'GZ_ICBC'
ORDER BY create_time DESC LIMIT 20;

-- 转账记录状态说明（status字段）：
-- 参考代码中的状态枚举值

-- 查询开户状态说明（account_status字段）：
-- 0=未开户, 1=正常, 2=冻结, 3=注销, 4=开户中, 5=失败
```
