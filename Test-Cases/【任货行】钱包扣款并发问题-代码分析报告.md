## 二、钱包操作层面的保护

### 2.1 钱包操作的统一锁机制

**实现方式**：使用 `@SyncLock` 注解 + AOP 切面

**涉及文件**：
- 注解定义：`SyncLock.java`
- 切面实现：`RedisLockAdvice.java`
- 业务实现：`WalletOperateBizServiceImpl.java`

**所有钱包操作方法都加了锁**：

```java
@Service
public class WalletOperateBizServiceImpl implements WalletOperateBizService {
    
    // 余额扣减
    @SyncLock(RedisConst.UPDATE_WALLET_ID_KEY)
    public Tuple3<Boolean, String, String> walletOperateBalanceSub(...) { }
    
    // 余额增加（充值）
    @SyncLock(RedisConst.UPDATE_WALLET_ID_KEY)
    public Tuple2<Boolean, String> walletOperateBalanceAdd(...) { }
    
    // 余额冻结
    @SyncLock(RedisConst.UPDATE_WALLET_ID_KEY)
    public Tuple2<Boolean, String> walletOperateBalanceFreez(...) { }
    
    // 余额解冻
    @SyncLock(RedisConst.UPDATE_WALLET_ID_KEY)
    public Tuple3<Boolean, String, String> walletOperateBalanceUnFreez(...) { }
    
    // ... 其他钱包操作方法
}
```

### 2.2 AOP 切面的锁机制

**切面执行顺序**（`RedisLockAdvice.java`）：

```java
@Order(-999)  // 优先级最高，最先执行
@Aspect
public class RedisLockAdvice {
    
    // 方法执行前：加锁
    @Before("pointCut()")
    public void addBeforeLogger(JoinPoint joinPoint) {
        String key = getLockKey(joinPoint, syncLock.value());
        RLock lock = redisLockUtil.lock(key);
        lock.lock(syncLock.LockTime(), syncLock.timeUnit());
        RedisLockHolder.setRlock(lock);  // 保存到 ThreadLocal
    }
    
    // 方法执行后：释放锁
    @AfterReturning("pointCut()")
    public void addAfterReturningLogger(JoinPoint joinPoint) {
        RLock lock = RedisLockHolder.getRlock();
        if (lock != null && lock.isLocked()) {
            lock.unlock();
        }
        RedisLockHolder.clean();
    }
    
    // 方法异常：释放锁
    @AfterThrowing(pointcut = "pointCut()", throwing = "ex")
    public void addAfterThrowingLogger(JoinPoint joinPoint, Exception ex) {
        RLock lock = RedisLockHolder.getRlock();
        if (lock != null && lock.isLocked()) {
            lock.unlock();
        }
        RedisLockHolder.clean();
    }
}
```

**关键点**：
- `@Order(-999)`：确保切面优先级最高
- ThreadLocal 保存锁对象：确保线程安全
- finally 块保证锁一定释放

## 三、完整的并发保护流程

### 3.1 账单扣款的完整调用链

```
balanceDeduct(deductBillId)
    ↓
【加账单锁】acquireBillLock(deductBillId)  ← 第一层保护：防止同一账单重复扣款
    ↓
doExecuteBalanceDeduct(deductRecord, truckUser)
    ↓
【加钱包锁】lock(WALLET_ID_KEY + walletId)  ← 第二层保护：防止同一钱包并发操作
    ↓
【开启事务】transactionTemplate.execute()
    ↓
batchSubWalletBalance(deductRecord, truckUser)
    ↓
walletOperateBalanceSub(subList)
    ↓
【AOP 加锁】@SyncLock 切面拦截              ← 第三层保护：钱包操作统一加锁
    ↓
【执行扣款】更新钱包余额 + 记录流水
    ↓
【AOP 释放锁】@AfterReturning 释放锁
    ↓
【提交事务】transaction commit
    ↓
【释放钱包锁】rLock.unlock()
    ↓
【释放账单锁】releaseLock(rLock, deductBillId)
```

### 3.2 并发场景下的执行顺序

**场景**：线程1扣款 + 线程2充值，同时操作同一钱包

```
时间轴：
T1: 线程1 - 获取账单锁成功
T2: 线程1 - 获取钱包锁成功
T3: 线程1 - 开启事务
T4: 线程1 - 执行扣款（余额 1000 → 800）
T5: 线程2 - 尝试获取钱包锁 → 【等待中】
T6: 线程1 - 提交事务（数据库更新完成）
T7: 线程1 - 释放钱包锁
T8: 线程2 - 获取钱包锁成功
T9: 线程2 - 读取余额（800元）← 正确的新值
T10: 线程2 - 执行充值（余额 800 → 1300）
T11: 线程2 - 提交事务
T12: 线程2 - 释放钱包锁
```

**关键**：线程2在T5时被阻塞，直到T7线程1释放锁且事务已提交，读到的是正确的新余额。

### 4.2 潜在风险点分析

**风险点1：双重加锁是否会死锁？**
- **分析**：不会死锁
- **原因1**：账单锁和钱包锁是不同的key
- **原因2**：加锁顺序固定：账单锁 → 钱包锁
- **原因3**：设置了超时时间（2-3分钟）

**风险点2：@SyncLock 与手动加锁冲突？**
- **分析**：可能存在冲突
- **现状**：`doExecuteBalanceDeduct` 方法手动加了钱包锁，调用的 `walletOperateBalanceSub` 也有 @SyncLock
- **影响**：同一线程对同一钱包加了两次锁
- **Redis Redisson锁特性**：支持可重入锁，同一线程可多次获取
- **结论**：不会有问题，但可能存在性能损耗

**风险点3：充值操作的并发保护**
- **分析**：充值方法 `walletOperateBalanceAdd` 也有 @SyncLock 注解
- **结论**：充值与扣款并发时，会被锁机制保护

**✓ 核心问题已修复**：
1. 账单扣款方法 `doExecuteBalanceDeduct` 已将锁提到事务之前

**✓ 多层保护机制**：
1. 第一层：账单级别锁（防止同一账单重复扣款）
2. 第二层：钱包级别锁（防止同一钱包并发操作）

**✓ 异常处理完善**：
- finally 块保证锁一定释放
- 可重入锁支持同一线程多次获取

### 6.2 存在的优化点

**优化点1：双重加锁**
- `doExecuteBalanceDeduct` 手动加锁 + `walletOperateBalanceSub` @SyncLock
- 建议：移除手动加锁，统一使用 @SyncLock，或移除 @SyncLock，统一手动加锁
- 原因：减少重复加锁的性能损耗

**优化点2：锁粒度**
- 当前锁的key是钱包ID，同一钱包的所有操作都会串行
- 建议：如果性能成为瓶颈，可考虑更细粒度的锁策略
