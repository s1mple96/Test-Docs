> 需求编号：1017536 | 提交ID：7f67240907 | 测试日期：2026-04-16

---

> ⚠️ **系统说明（防混淆）**
>
> 本次改动代码全部在 **`java/fenmi/` 分米ETC系统**（`fenmi-etc-job` 模块），操作的数据库是 **`fenmi_etc`**，与独立的任通行系统（`java/rtx/`，`rtx` 库）无关。
>
> 需求中"任通行"指的是分米ETC平台内 **苏通卡运营商（`operator_code = 'TXB'`）** 的用户；"新发行"指 **蒙通卡运营商（`operator_code = 'MTK'`）** 的用户。两者均属于分米ETC系统，数据均在 `fenmi_etc` 库。

## 关键字段需求映射

| 需求原文描述 | 对应库.表.字段 | 说明 |
|-------------|--------------|------|
| 扣款状态=扣款失败 | `fenmi_etc.etc_deduct_record.status = 2` | TollBillStatusEnum.PAY_FAILT = 2 |
| 创建时间=近3天（含当天） | `fenmi_etc.etc_deduct_record.create_time >= now() - 3天` | 代码实现为 `LocalDateTime.now().minusDays(3)`（72小时滑动窗口，非自然日，需与产品确认语义） |
| 最近一次扣款时间 | `fenmi_etc.etc_deduct_record.deduct_time` | 每次发起扣款动作时更新，代扣/补扣/轮扣均更新此字段 |
| 排除最近20分钟内有扣款动作 | `fenmi_etc.etc_deduct_record.deduct_time <= now() - 20分钟` | SQL 条件：`deduct_time <= #{endDeductTime}`，即只捞最后一次扣款超过20分钟的数据 |
| 任通行（钱包1.0用户） | `fenmi_etc.etc_deduct_record.operator_code = 'TXB'`，`fenmi_etc.etc_wallet_deduction_method.deduction_method IN (1,2)`（银行卡/钱包） | ⚠️ 此处"任通行"= 分米ETC平台内苏通卡（TXB）用户，数据在 **fenmi_etc 库**，与独立任通行系统（rtx 库）无关 |
| 任通行（钱包2.0代扣用户） | `fenmi_etc.etc_deduct_record.operator_code = 'TXB'`，`fenmi_etc.etc_wallet_deduction_method.deduction_method = 3`（微信代扣） | 同上，微信代扣通道，DeductionMethodEnum.WX = 3 |
| 新发行，所有用户 | `fenmi_etc.etc_deduct_record.operator_code = 'MTK'` | 分米ETC平台内蒙通卡（MTK）用户，数据同在 **fenmi_etc 库** |
| 补扣类型 | `BillDeductTypeEnum.FAIL_SUPPLEMENTARY_DEDUCT`（value = failSupplementaryDeduct） | 新增枚举值，与 FAIL_DEDUCT / FIRST_DEDUCT 区分 |
| 钱包补扣定时任务 | `BillWithholdDeductTask.failtBillDeductTask()` → `billRecordFailtReDeduct()` | 原有任务，此次改动新增了20分钟过滤 |

## 1. 新增补扣任务 — 基本功能

### 1.1 正常补扣流程

#### TC-001 苏通卡（TXB，分米ETC内的"任通行"用户）扣款失败账单正常触发补扣

- **前置条件**:
  - 系统：分米ETC定时任务服务（fenmi-etc-job）
  - 数据库：fenmi_etc
  - 造数 SQL（在测试环境执行）：
    ```sql
    -- 确认存在满足条件的账单（扣款失败 + 近3天 + 最后扣款超过20分钟）
    SELECT id, status, deduct_time, create_time, operator_code
    FROM fenmi_etc.etc_deduct_record
    WHERE operator_code = 'TXB'
      AND status = 2
      AND create_time >= DATE_SUB(NOW(), INTERVAL 3 DAY)
      AND deduct_time <= DATE_SUB(NOW(), INTERVAL 20 MINUTE)
    LIMIT 5;
    ```
  - 若无，手动插入一条测试账单（status=2，create_time=当前时间，deduct_time=当前时间-30分钟，operator_code='TXB'）
  - 当前时间在 06:00–18:00 之间（任务执行时间窗口需求：「每日在6点-18点执行」）

- **测试步骤**:
  1. 在 Jenkins 或调度平台手动触发 `billWithholdDeductTask.failSupplementaryDeductTask` 任务
  2. 观察 fenmi-etc-job 日志，确认打印"失败补扣扣款任务开始"
  3. 查询该账单状态变化：
     ```sql
     SELECT id, status, deduct_time, update_time
     FROM fenmi_etc.etc_deduct_record
     WHERE id = {测试账单ID} LIMIT 1;
     ```
  4. 确认日志中出现该账单的扣款动作（deductRecordId）

- **预期结果**:
  - 日志出现"失败补扣扣款任务开始"和"失败补扣扣款任务结束"
  - 该账单被捞取并发起扣款（日志中有 `FAIL_SUPPLEMENTARY_DEDUCT` 类型的扣款记录）
  - `deduct_time` 更新为当前时间；账单状态根据扣款结果更新（成功→1，失败→2，支付中→3）

## 2. 过滤条件验证

### 2.1 创建时间近3天过滤

#### TC-005 创建时间恰好等于3天前的账单 — 应被捞取

- **前置条件**:
  - 构造账单：`create_time = DATE_SUB(NOW(), INTERVAL 3 DAY)`（近似72小时，边界值），status=2，deduct_time < now()-20分钟
  - 注意：代码实现为 `LocalDateTime.now().minusDays(3)`（72小时滑动窗口），非自然日0点起算（需求：「含当天」语义待与产品确认）

- **测试步骤**:
  1. 造数并触发任务
  2. 查询日志确认该账单ID是否出现在处理记录中

- **预期结果**:
  - 该账单被捞取并发起补扣（`create_time = now()-3天` 应满足 `create_time >= now()-3天` 条件，处于临界）

## 4.1bis 客车原有失败重扣任务（failtBillDeductTask）— 新增20分钟筛选专项

- **预期结果**:
  - 账单未被捞取，不发起 `FAIL_DEDUCT` 扣款
  - `deduct_time`、`status` 均无变化（需求：「钱包补扣定时任务改动点：排除最近一次扣款时间在最近20分钟内的数据，不执行补扣下黑」）

### 4.2 防止与轮扣并发冲突
