# 【任通行】快速缴费充值日志监控测试用例

## 1. 需求背景

### 1.1 需求概述
任通行新增还款/充值H5功能上线后，由于与原有还款逻辑共用同一接口，无法准确统计从快速缴费入口进入的用户数量及成功率。为实现业务数据监控和分析，需要在关键节点增加日志上传功能。

### 1.2 需求描述
1. 通过快速入口或链接进入"路费还款"页面，输入完车牌查询成功后，上传日志（公共接口）
   - 日志内容：{phone: 接口返回手机号, req: 请求参数对象-json字符串, source: '快速缴费充值进入'}

2. 支付成功后（微信支付、银行卡支付），上传日志（公共接口）
   - 日志内容：{phone: 上一次手机号, req: 请求参数对象-json字符串, source: '快速缴费充值成功'}

3. 卡签检测未搜到设备时，上传日志（公共接口）
   - 日志内容：{phone: 用户手机号, req: 检测请求参数-json字符串, source: '卡签检测未搜到设备'}

### 1.3 测试范围
- 快速还款页面日志上传
- 支付成功日志上传
- 卡签检测异常日志上传
- 日志内容完整性和准确性验证
- 不同支付方式下的日志记录

## 2. 车牌查询成功日志上传测试

### 2.1 快速入口查询成功测试

#### TC-001 通过快速入口输入车牌查询成功日志上传
- **前置条件**：用户通过快速缴费入口进入路费还款页面
- **测试步骤**：
  1. 点击快速缴费充值入口链接进入还款页面
  2. 在车牌输入框输入有效车牌号（如：粤A12345）
  3. 点击查询按钮触发查询接口
  4. 接口返回查询成功结果（包含手机号、欠费信息等）
  5. 验证系统是否自动调用公共日志上传接口
- **预期结果**：系统自动上传日志，日志内容包含{phone: "138****1234", req: "{carNum:'粤A12345',...}", source: "快速缴费充值进入"}

#### TC-002 通过分享链接进入查询成功日志上传
- **前置条件**：用户通过分享的H5链接进入路费还款页面
- **测试步骤**：
  1. 通过分享链接打开路费还款H5页面
  2. 输入有效车牌号并点击查询
  3. 查询接口返回成功结果
  4. 验证日志上传接口的调用
  5. 检查日志内容中的source字段值
- **预期结果**：日志上传成功，source字段正确标识为"快速缴费充值进入"，请求参数完整记录

#### TC-003 查询返回多条欠费记录日志上传
- **前置条件**：车牌号存在多条未缴费账单
- **测试步骤**：
  1. 通过快速入口进入还款页面
  2. 输入有多条欠费记录的车牌号
  3. 查询接口返回多条账单信息
  4. 验证日志上传的请求参数是否包含所有账单信息
  5. 检查phone字段是否正确脱敏
- **预期结果**：日志成功上传，req字段包含完整的查询参数和返回的多条账单信息，phone字段正确脱敏

#### TC-004 首次查询成功后再次查询日志上传
- **前置条件**：用户已完成一次查询
- **测试步骤**：
  1. 完成首次车牌查询
  2. 返回修改车牌号重新查询
  3. 第二次查询成功
  4. 验证是否重复上传日志
  5. 检查两次日志的req参数是否区分
- **预期结果**：每次查询成功都上传日志，每条日志的req参数准确反映对应的查询请求

### 2.2 查询失败场景日志处理测试

#### TC-005 车牌号不存在查询失败不上传日志

#### TC-006 车牌号无欠费记录查询成功日志上传
- **前置条件**：车牌号存在但无欠费记录
- **测试步骤**：
  1. 输入无欠费记录的有效车牌号
  2. 查询接口返回成功但欠费列表为空
  3. 验证是否上传日志
  4. 检查日志中的req参数是否包含空账单列表信息
- **预期结果**：查询接口成功返回时上传日志，即使欠费列表为空

#### TC-007 查询接口超时异常不上传日志
- **前置条件**：模拟网络延迟或接口响应超时
- **测试步骤**：
  1. 输入车牌号点击查询
  2. 接口响应超时触发前端超时处理
  3. 页面显示网络异常提示
  4. 验证是否调用日志上传接口
- **预期结果**：接口异常不上传日志，避免脏数据

### 2.3 日志内容完整性测试

#### TC-008 日志phone字段脱敏验证
- **前置条件**：查询接口返回完整手机号
- **测试步骤**：
  1. 完成车牌查询获取手机号信息
  2. 触发日志上传
  3. 拦截日志上传请求查看phone字段内容
  4. 验证手机号是否按规则脱敏（中间4位星号）
  5. 确认原始手机号未被明文上传
- **预期结果**：phone字段正确脱敏为"138****1234"格式，保护用户隐私

#### TC-009 日志req字段JSON格式验证
- **前置条件**：完成车牌查询触发日志上传
- **测试步骤**：
  1. 拦截日志上传请求
  2. 检查req字段是否为标准JSON字符串格式
  3. 验证JSON字符串是否可正常解析
  4. 确认关键参数（车牌号、查询时间等）是否完整
  5. 检查特殊字符是否正确转义
- **预期结果**：req字段为合法JSON字符串，包含完整的请求参数信息，特殊字符正确转义

#### TC-010 日志source字段固定值验证
- **前置条件**：通过快速入口完成查询
- **测试步骤**：
  1. 多次执行车牌查询操作
  2. 拦截每次的日志上传请求
  3. 验证source字段值的一致性
  4. 确认固定为"快速缴费充值进入"
  5. 检查字段值无拼写错误
- **预期结果**：source字段恒定为"快速缴费充值进入"，无任何变化

## 3. 支付成功日志上传测试

  2. 选择需要支付的账单

  4. 完成微信支付流程（支付成功）

- **预期结果**：支付成功后自动上传日志，日志内容包含{phone: "138****1234", req: "{支付参数...}", source: "快速缴费充值成功"}

#### TC-012 微信支付成功日志phone字段延续性验证

- **测试步骤**：
  1. 完成车牌查询（上传第一条日志）

  3. 拦截支付成功日志

  5. 确认phone字段为同一个手机号
- **预期结果**：支付成功日志中的phone字段与查询日志中的phone字段一致，保持数据关联性

#### TC-013 微信支付成功日志req字段内容验证
- **前置条件**：微信支付流程已完成
- **测试步骤**：
  1. 完成微信支付

  3. 解析req字段的JSON内容
  4. 验证是否包含支付金额、订单号、支付方式等关键信息
  5. 检查数据格式和精度
- **预期结果**：req字段包含完整的支付请求参数，包括订单信息、支付金额、支付方式等

#### TC-014 微信支付部分成功日志上传

  1. 选择多笔账单进行支付

  3. 验证日志上传时机
  4. 检查req参数中是否区分成功和失败的账单
  5. 确认日志上传次数
- **预期结果**：支付成功的账单触发日志上传，req参数准确反映支付成功的账单信息
