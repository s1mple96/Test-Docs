# 【任货行】钱包扣款并发问题 - 测试工程师操作指南

## 一、理解问题本质

### 1.1 什么是事务和锁？

**简单理解**：
- **事务**：一组数据库操作，要么全部成功，要么全部失败回滚
- **锁**：防止多个操作同时修改同一数据，类似排队机制

**类比**：
```
银行转账场景：
事务 = 从A账户扣钱 + 给B账户加钱（必须同时成功）
锁 = 操作A账户时，别人不能同时操作A账户
```

### 1.2 BUG原因

**旧逻辑（错误）**：
```
线程1：开启事务 → 加锁 → 扣款 → 释放锁 → 提交事务
                    ↑                ↓
                    锁的范围          事务还未提交
                                    ↓
线程2：                        加锁成功 → 读到旧余额 → 计算错误
```

**问题**：锁已释放，但事务未提交，数据还未真正更新，导致其他线程读到旧数据。

**新逻辑（正确）**：
```
线程1：加锁 → 开启事务 → 扣款 → 提交事务 → 释放锁
       ↑                               ↓
       锁的范围一直持续到事务提交完成
                                      ↓
线程2：                              等待锁 → 读到新余额 → 计算正确
```

## 二、测试思路

### 2.1 测试核心目标

**验证并发安全性**：
1. 同一钱包同时扣款+充值，余额计算正确
2. 流水记录完整，不丢失
3. 余额 = 初始余额 + 充值总额 - 扣款总额

**难点1：如何制造并发？**
- 单个操作太快，无法真正并发
- 需要工具或脚本同时发起多个请求

**难点2：如何验证结果？**
- 查询钱包余额
- 查询钱包流水记录
- 手工计算并对比

### 4.2 并发测试（核心）

#### 方法一：使用 Postman（推荐新手）

**工具**：Postman

**步骤**：

1. **创建Collection**
   - 新建 Collection："钱包并发测试"
   - 添加请求1：账单扣款接口
   - 添加请求2：钱包充值接口

2. **配置请求参数**
   - 请求1参数：`wallet_id`, `bill_id`, `amount=200`
   - 请求2参数：`wallet_id`, `recharge_amount=500`

3. **使用 Collection Runner**
   - 选择 Collection："钱包并发测试"
   - 设置 Iterations：1次
   - 设置 Delay：0ms（同时发起）
   - 点击 Run

4. **验证结果**（参考4.3节）

#### 方法二：使用 JMeter（推荐熟练测试）

**工具**：Apache JMeter

**步骤**：

1. **创建测试计划**
   - 添加线程组1：账单扣款
     - 线程数：5
     - Ramp-Up：0秒
     - 循环次数：1
   - 添加线程组2：钱包充值
     - 线程数：5
     - Ramp-Up：0秒
     - 循环次数：1

2. **添加HTTP请求**
   - 线程组1 → HTTP请求 → 账单扣款接口
   - 线程组2 → HTTP请求 → 钱包充值接口

3. **配置参数**
   - 使用同一个 `wallet_id`
   - 不同的 `bill_id` 或 `order_id`

4. **添加监听器**
   - 查看结果树
   - 聚合报告

5. **执行测试**
   - 点击启动按钮
   - 等待所有请求完成

6. **验证结果**（参考4.3节）

#### 方法三：使用Python脚本（推荐自动化）

**工具**：Python + requests + threading

**脚本示例**：
```python
import requests
import threading
import time

# 测试配置
WALLET_ID = "测试钱包ID"
BASE_URL = "http://测试环境地址"
DEDUCT_URL = f"{BASE_URL}/api/deduct"
RECHARGE_URL = f"{BASE_URL}/api/recharge"

# 扣款函数
def deduct_bill(bill_id, amount):
    try:
        response = requests.post(DEDUCT_URL, json={
            "wallet_id": WALLET_ID,
            "bill_id": bill_id,
            "amount": amount
        })
        print(f"扣款 {bill_id}: {response.status_code} - {response.text}")
    except Exception as e:
        print(f"扣款 {bill_id} 异常: {e}")

# 充值函数
def recharge_wallet(order_id, amount):
    try:
        response = requests.post(RECHARGE_URL, json={
            "wallet_id": WALLET_ID,
            "order_id": order_id,
            "amount": amount
        })
        print(f"充值 {order_id}: {response.status_code} - {response.text}")
    except Exception as e:
        print(f"充值 {order_id} 异常: {e}")

# 并发测试
threads = []

# 创建5个扣款线程
for i in range(5):
    t = threading.Thread(target=deduct_bill, args=(f"BILL_{i}", 100))
    threads.append(t)

# 创建5个充值线程
for i in range(5):
    t = threading.Thread(target=recharge_wallet, args=(f"ORDER_{i}", 200))
    threads.append(t)

# 同时启动所有线程
for t in threads:
    t.start()

# 等待所有线程完成
for t in threads:
    t.join()

print("并发测试完成")
```

**执行**：
```bash
python concurrent_test.py
```

### 4.3 结果验证（最重要）

#### Step 1：记录测试前的初始状态
```sql
-- 查询初始余额
SELECT wallet_id, blance AS initial_balance, update_time 
FROM hcb.hcb_monthly_paylater_wallet 
WHERE wallet_id = '测试钱包ID';

-- 记录：initial_balance = 1000元
```

#### Step 2：执行并发测试
- 按照4.2节的方法执行并发测试
- 假设：5次扣款（每次100元）+ 5次充值（每次200元）

#### Step 3：查询最终余额
```sql
-- 查询最终余额
SELECT wallet_id, blance AS final_balance, update_time 
FROM hcb.hcb_monthly_paylater_wallet 
WHERE wallet_id = '测试钱包ID';

-- 预期：final_balance = 1000 - 500 + 1000 = 1500元
```

#### Step 4：查询流水记录
```sql
-- 查询所有流水记录
SELECT 
  id, trade_amount, before_amount, surplus_amount, 
  fund_direction, biz_type, create_time 
FROM hcb.hcb_monthly_paylater_wallet_flow 
WHERE wallet_id = '测试钱包ID' 
  AND create_time >= '并发测试开始时间'
ORDER BY create_time;
```

#### Step 5：统计流水金额
```sql
-- 统计充值和扣款总额
SELECT 
  SUM(CASE WHEN fund_direction = 'IN' THEN trade_amount ELSE 0 END) AS total_recharge,
  SUM(CASE WHEN fund_direction = 'OUT' THEN trade_amount ELSE 0 END) AS total_deduct,
  COUNT(*) AS total_count
FROM hcb.hcb_monthly_paylater_wallet_flow 
WHERE wallet_id = '测试钱包ID' 
  AND create_time >= '并发测试开始时间';

-- 预期：total_recharge = 1000, total_deduct = 500, total_count = 10
```

#### Step 6：验证一致性
```
计算公式：
final_balance = initial_balance + total_recharge - total_deduct

实际计算：
final_balance = 1000 + 1000 - 500 = 1500元

与数据库查询的 final_balance 比对：
✓ 一致 = 测试通过
✗ 不一致 = 测试失败（存在并发问题）
```

#### Step 7：检查流水连续性
```sql
-- 检查流水的 before_amount 和 surplus_amount 是否连续
SELECT 
  id,
  before_amount,
  trade_amount,
  surplus_amount,
  fund_direction,
  create_time,
  LAG(surplus_amount) OVER (ORDER BY create_time) AS prev_surplus
FROM hcb.hcb_monthly_paylater_wallet_flow 
WHERE wallet_id = '测试钱包ID' 
  AND create_time >= '并发测试开始时间'
ORDER BY create_time;

-- 预期：每条流水的 before_amount = 上一条流水的 surplus_amount
-- 如果不连续，说明存在并发读取旧值的问题
```

#### Step 8：检查是否有流水丢失
```
实际流水数量 = 数据库查询的 total_count
预期流水数量 = 扣款次数 + 充值次数 = 5 + 5 = 10

✓ 相等 = 流水完整
✗ 不相等 = 流水丢失（严重问题）
```

### 4.4 判断标准

**测试通过**：
1. ✓ 最终余额 = 初始余额 + 充值总额 - 扣款总额
2. ✓ 流水记录数量 = 实际操作次数
3. ✓ 流水的 before_amount 和 surplus_amount 连续
4. ✓ 所有账单状态正确更新
5. ✓ 无异常日志或错误

**测试失败**：
1. ✗ 最终余额计算不一致（数据覆盖问题）
2. ✗ 流水记录丢失（并发写入失败）
3. ✗ 流水金额断层（多个线程读到相同的旧值）
4. ✗ 账单状态未更新（事务回滚）

## 六、常见问题

### Q1：我不懂代码，怎么测试事务和锁？
**A**：不需要懂代码实现，只需要：
1. 会发起接口请求（Postman）
2. 会查询数据库（SQL）
3. 会计算余额（加减法）
4. 会对比结果

### Q2：如何同时发起多个请求？
**A**：三种方法：
1. Postman Collection Runner（最简单）
2. JMeter 线程组（专业工具）
3. Python 脚本（自动化）

### Q3：如何判断测试通过还是失败？
**A**：核心公式：
```
最终余额 = 初始余额 + 充值总额 - 扣款总额
```
计算结果与数据库余额一致 = 通过

### Q4：如果测试失败，怎么办？
**A**：
1. 截图保存测试数据
2. 导出流水记录
3. 记录测试步骤
4. 提交BUG给开发

### Q5：需要多少测试数据？
**A**：建议准备：
- 5-10个测试钱包账户
- 每个钱包10-20个待扣款账单
- 钱包余额设置为5000-10000元

### Q6：测试环境会影响生产环境吗？
**A**：不会，测试环境独立，但注意：
1. 只在测试环境执行
2. 使用测试数据
3. 不要操作真实用户数据
