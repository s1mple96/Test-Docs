# 【ETC系统】新发行欠费下黑覆盖无扣款方式下黑测试用例

## 1. 原有下黑业务功能回归测试

### 1.1 基础功能验证测试

#### TC-001 普通欠费用户正常下黑测试
- **前置条件**：数据库中存在用户状态为正常（status=1或8），黑名单状态为0（正常），有扣款失败记录且失败时间超过配置的下黑时长（如1小时）
- **测试步骤**：
  1. 查询正常用户，准备测试数据：blacklist_status=0，status=1，operator_code=TXB
  2. 准备扣款失败记录：将用户的账单设置为扣款失败状态（status=2），账单创建时间设置为3小时前
  3. 更新用户欠费信息：设置debt_count=1，debt_amt=10.50，debt_time为3小时前
  4. 登录后台管理系统，进入系统监控-定时任务页面
  5. 搜索任务ID为47的"对用户进行下黑操作任务"，点击"执行一次"按钮
  6. 执行完成后查询用户表验证字段更新：blacklist_status、black_source、debt_count、debt_amt
- **预期结果**：普通欠费用户成功下黑，blacklist_status=1，black_source=1（欠费下黑），欠款信息正确记录

#### TC-002 已欠费下黑用户不重复处理测试
- **前置条件**：用户blacklist_status=1，black_source=1（欠费下黑）
- **测试步骤**：
  1. 查询已经是欠费下黑状态的用户：blacklist_status=1且black_source=1
  2. 确认用户有符合条件的扣款失败记录
  3. 执行任务ID为47的"对用户进行下黑操作任务"
  4. 查看任务执行日志，检查该用户是否被捞取
  5. 查询用户表验证black_times等字段是否有变化
- **预期结果**：已经是欠费下黑的用户不会被重复捞取和处理，black_times等字段保持不变

#### TC-003 用户正在扣款中场景不下黑测试
- **前置条件**：用户blacklist_status=0，有扣款记录处于"扣款中"状态
- **测试步骤**：
  1. 准备测试用户：符合下黑时间条件，blacklist_status=0
  2. 设置用户的扣款记录状态为"扣款中"
  3. 执行任务ID为47的"对用户进行下黑操作任务"
  4. 查看任务执行日志，搜索该用户的处理记录
  5. 验证日志中是否输出"用户正在扣款中，无法下黑"的提示
  6. 查询用户表确认blacklist_status仍为0
- **预期结果**：用户正在扣款中时不下黑，日志输出"用户正在扣款中，无法下黑"，用户状态保持不变

#### TC-004 账单未超过时间阈值不下黑测试
- **前置条件**：运营商配置black_time=1小时，用户有扣款失败记录但创建时间为30分钟前
- **测试步骤**：
  1. 查询运营商下黑配置表确认TXB的black_time=1小时
  2. 准备测试用户：blacklist_status=0，status=1
  3. 设置扣款失败记录：status=2，create_time为当前时间减去30分钟
  4. 执行任务ID为47的"对用户进行下黑操作任务"
  5. 查看任务执行日志，检查该用户是否被下黑
  6. 查询用户表验证blacklist_status是否仍为0
- **预期结果**：账单创建时间未超过配置的1小时阈值，用户不下黑，blacklist_status保持为0

#### TC-005 账单超过时间阈值触发下黑测试
- **前置条件**：运营商配置black_time=1小时，用户有扣款失败记录且创建时间为2小时前
- **测试步骤**：
  1. 查询运营商下黑配置表确认TXB的black_time=1小时
  2. 准备测试用户：blacklist_status=0，status=1
  3. 设置扣款失败记录：status=2，create_time为当前时间减去2小时
  4. 更新用户欠费信息：debt_count=1，debt_amt=10.50，debt_time为2小时前
  5. 执行任务ID为47的"对用户进行下黑操作任务"
  6. 查询用户表验证下黑结果
- **预期结果**：账单创建时间超过1小时阈值，用户成功下黑，blacklist_status=1，black_source=1

#### TC-006 欠费金额累加计算准确性测试
- **前置条件**：用户有3条扣款失败记录，total_fee分别为10元、20元、30元
- **测试步骤**：
  1. 准备测试用户，创建3条扣款失败记录
  2. 设置第1条记录：total_fee=10.00，status=2，create_time为3小时前
  3. 设置第2条记录：total_fee=20.00，status=2，create_time为2小时前
  4. 设置第3条记录：total_fee=30.00，status=2，create_time为1小时前
  5. 执行任务ID为47的"对用户进行下黑操作任务"
  6. 查询用户表验证debt_count和debt_amt字段
- **预期结果**：欠费金额使用BigDecimal准确累加，debt_count=3，debt_amt=60.00

### 1.2 异常场景和边界值测试

#### TC-008 配置black_time为0时使用默认值测试
- **前置条件**：运营商下黑配置表中black_time=0
- **测试步骤**：
  1. 更新运营商下黑配置：将TXB的black_time设置为0
  2. 准备测试用户：blacklist_status=0，有1小时前的扣款失败记录

  1. 准备测试用户：blacklist_status=0，black_times=2

  4. 等待两次任务执行完成
  5. 查询用户表验证black_times字段的值
- **预期结果**：Redis分布式锁生效，用户只被下黑一次，black_times=3（不是4）

## 2. 新功能：无扣款方式下黑用户覆盖测试

- **预期结果**：black_source=5的用户被成功捞取，出现在任务处理日志中

#### TC-011 批量无扣款方式下黑用户捞取测试
- **前置条件**：数据库中存在5个blacklist_status=1且black_source=5的用户

  1. 准备5个测试用户：全部设置blacklist_status=1，black_source=5

#### TC-012 无扣款方式下黑但无欠费账单不处理测试

#### TC-013 混合场景：正常用户和无扣款方式用户同时处理测试
- **前置条件**：数据库中同时存在blacklist_status=0的普通用户和blacklist_status=1且black_source=5的用户
- **测试步骤**：
  1. 准备3个普通用户：blacklist_status=0，black_source=0

- **预期结果**：两类用户都被成功捞取和处理，最终black_source都为1（欠费下黑）
