# 【任通行】免登录小程序还款解黑功能测试用例

## 测试背景

**需求描述**：业务端启动客车催收功能，为客服提供配套工具。有意愿还款的客户，可通过小程序发送还款链接，用户点击后免登录快速还款。

**页面说明**：在原有的任通行微信小程序中新增免登录还款页面（非H5页面，是小程序原生页面）

**涉及范围**：
1. 新增：免登录小程序还款页面
2. 回归：登录后还款解黑功能
3. 重点：三种钱包模式（钱包1.0、钱包2.0余额、钱包2.0代扣）的下黑/解黑逻辑

**系统模块**：
- 任通行微信小程序（新增免登录还款页面）
- 任通行后台管理系统
- 定时任务系统

### 1.4 免登录充值与三种钱包模式测试

> **重要说明**：虽然充值功能本身相同（都是增加余额），但不同钱包模式的用户在充值时可能有不同的业务处理，特别是钱包2.0代扣模式充值后可能触发自动代扣支付账单。

#### TC-016 免登录充值-钱包1.0模式
- **前置条件**：
  - 用户钱包模式：wallet_mode = '1', wallet_deduction = '1'
  - 账户余额：-50元
- **测试步骤**：
  1. 通过免登录方式输入车牌号查询
  2. 显示欠费50元
  3. 充值100元
  4. 验证微信支付成功
  5. 查询钱包余额
  6. 查询rtx_etc_balance_flow表
- **预期结果**：
  - 充值成功，余额从-50元变为50元
  - 流水记录正确（flow_type、balance_before、balance_after）
  - **钱包1.0特点**：充值后余额直接增加，无其他自动操作

#### TC-016-1 免登录充值-钱包2.0余额模式
- **前置条件**：
  - 用户钱包模式：wallet_mode = '2', wallet_deduction = '1'
  - 账户余额：-80元
- **测试步骤**：
  1. 通过免登录方式充值100元
  2. 验证充值成功
  3. 查询钱包余额
  4. 查询充值流水
- **预期结果**：
  - 充值成功，余额从-80元变为20元
  - **钱包2.0余额特点**：充值逻辑与钱包1.0相同
  - 流水记录wallet_mode = '2', wallet_deduction = '1'

#### TC-016-2 免登录充值-钱包2.0代扣模式（重点）
- **前置条件**：
  - 用户钱包模式：wallet_mode = '2', wallet_deduction = '2'
  - 账户余额：-50元
  - 有2笔欠费账单（pay_status = '3'，每笔15元）
- **测试步骤**：
  1. 通过免登录方式充值100元
  2. 验证充值成功
  3. 立即查询账单状态（pay_status）
  4. 查询钱包余额
  5. 等待定时任务执行代扣（每天6点执行）
  6. 再次查询账单状态和余额
- **预期结果**：
  - 充值成功，余额从-50元变为50元 ✅
  - ⚠️ **钱包2.0代扣特点**：充值后余额增加，但账单支付由定时任务处理
  - 充值后立即查询：
    - 账单状态仍为'3'（欠费）
    - 余额为50元
  - 定时任务执行后：
    - 2笔账单的pay_status变为'1'（已支付）
    - 余额从50元扣除30元（2×15元），变为20元
    - 有代扣流水记录

#### TC-016-3 三种钱包模式充值对比测试
- **前置条件**：准备3个测试用户，分别对应三种钱包模式，初始余额都为-50元
- **测试数据**：
  | 用户 | 钱包模式 | wallet_mode | wallet_deduction | 欠费账单 |
  |-----|---------|-------------|------------------|---------|
  | 用户A | 钱包1.0 | '1' | '1' | 0笔 |
  | 用户B | 钱包2.0余额 | '2' | '1' | 0笔 |
  | 用户C | 钱包2.0代扣 | '2' | '2' | 2笔（共30元）|
- **测试步骤**：
  1. 三个用户都通过免登录方式充值100元
  2. 立即查询：对比充值后的余额变化、流水记录、账单状态
  3. 等待定时任务执行（每天6点）
  4. 再次查询：对比余额变化、账单状态变化
- **预期结果**：
  - **充值后立即查询**：
    - 用户A：余额变为50元，无欠费账单
    - 用户B：余额变为50元，无欠费账单
    - 用户C：余额变为50元，账单状态仍为'3'（欠费）
  - **定时任务执行后查询**：
    - 用户A：余额仍为50元（无欠费账单，不扣款）
    - 用户B：余额仍为50元（无欠费账单，不扣款）
    - 用户C：余额变为20元（从余额或银行卡代扣支付了30元账单），账单变为已支付（pay_status='1'）
  - **关键差异**：三种钱包模式的账单支付都由定时任务处理，差异在于扣款方式（余额扣款 vs 银行卡代扣）

### 测试核心逻辑 ✅

```
充值层面（分层测试）
├─ 登录充值 × 3种钱包模式      ← 测试充值功能本身
│  ├─ 钱包1.0                 ← 余额直接增加
│  ├─ 钱包2.0余额             ← 余额直接增加
│  └─ 钱包2.0代扣             ← 余额直接增加（账单支付等定时任务）⚠️
│
├─ 免登录充值 × 3种钱包模式    ← 测试充值功能本身
   ├─ 钱包1.0                 ← 余额直接增加
   ├─ 钱包2.0余额             ← 余额直接增加
   └─ 钱包2.0代扣             ← 余额直接增加（账单支付等定时任务）⚠️
   
   ↓ 充值成功后余额变化

账单支付层面（定时任务处理，每日6-20点执行）
├─ 钱包1.0      → 从钱包余额扣款或银行卡代扣
├─ 钱包2.0余额  → 从钱包余额扣款
└─ 钱包2.0代扣  → 从银行卡代扣 ⚠️

解黑层面（定时任务判断）
├─ 钱包1.0      → balance >= 0 解黑
├─ 钱包2.0余额  → balance >= 0 解黑
└─ 钱包2.0代扣  → 无欠费账单 解黑（不看余额）⚠️

⚠️ 充值、账单代扣、下黑/解黑 都由定时任务处理，与充值方式（登录/免登录）无关
```

**关键认知**：
- ✅ **充值功能：三种钱包模式充值逻辑完全相同**，都只是余额直接增加
- ✅ **账单支付：三种钱包模式都由定时任务处理**（每日6-20点执行），不是充值后立即触发
- ✅ **三种模式的差异**：差异在于账单支付方式（余额扣款 vs 银行卡代扣），不在于触发时机
- ✅ **登录/免登录都要测试3种钱包模式**：确保不同充值入口在不同钱包模式下都正确处理
- ✅ **下黑/解黑由定时任务执行**：与充值方式无关，只与钱包模式和账户状态有关
