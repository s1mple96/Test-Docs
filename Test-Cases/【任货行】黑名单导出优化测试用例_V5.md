# 【任货行】黑名单导出优化测试用例

## 1. 归档统计任务测试

### 1.1 归档任务基本功能测试

#### TC-001 执行归档统计任务
- **前置条件**：MongoDB流水表有黑名单用户的历史流水数据
- **测试步骤**：
  1. 清空统计表验证初始状态
  2. 调用归档任务接口`/test/etcCardBalanceFlowStatistics`
  3. 等待任务执行完成
  4. 查询MongoDB统计表`hcb_etc_card_balance_flow_statistics`
- **预期结果**：统计表成功生成用户记录，包含lateFee、lawyerFee、monthFee、lateFeeRefund、blackTime、status字段，金额准确

#### TC-002 验证滞纳金归档准确性
- **前置条件**：归档任务已执行，用户有OPERATOR_TYPE=11的流水数据
- **测试步骤**：
  1. 从统计表取一个用户的truckUserWalletId
  2. 手工统计该用户MongoDB流水表中OPERATOR_TYPE=11的AMOUNT总和
  3. 对比统计表的lateFee字段值
- **预期结果**：lateFee与手工统计结果一致

#### TC-003 验证律师诉讼费归档准确性
- **前置条件**：归档任务已执行，用户有OPERATOR_TYPE=71的流水数据
- **测试步骤**：
  1. 从统计表取一个用户
  2. 手工统计该用户OPERATOR_TYPE=71的AMOUNT总和
  3. 对比统计表的lawyerFee字段值
- **预期结果**：lawyerFee与手工统计结果一致

#### TC-004 验证月管理费归档准确性
- **前置条件**：归档任务已执行，用户有OPERATOR_TYPE=100的流水数据
- **测试步骤**：
  1. 从统计表取一个用户
  2. 手工统计该用户OPERATOR_TYPE=100的AMOUNT总和
  3. 对比统计表的monthFee字段值
- **预期结果**：monthFee与手工统计结果一致

#### TC-005 验证滞纳金退费归档准确性
- **前置条件**：归档任务已执行，用户有OPERATOR_TYPE=108的流水数据
- **测试步骤**：
  1. 从统计表取一个用户
  2. 手工统计该用户OPERATOR_TYPE=108的AMOUNT总和
  3. 对比统计表的lateFeeRefund字段值
- **预期结果**：lateFeeRefund与手工统计结果一致

#### TC-006 验证下黑时间超过3个月的用户归档
- **前置条件**：准备两类用户，用户A下黑时间超过3个月且有MongoDB流水，用户B下黑时间在3个月内仅有MySQL流水
- **测试步骤**：
  1. 清空统计表
  2. 执行归档任务
  3. 查询统计表验证归档结果
- **预期结果**：用户A生成统计记录，用户B不生成记录

#### TC-007 验证根据blackTime年份选择MongoDB表
- **前置条件**：准备用户A的blackTime为2024年，流水在hcb_etccardbalanceflow_backup_2024表，用户B的blackTime为2023年，流水在hcb_etccardbalanceflow_backup主表
- **测试步骤**：
  1. 执行归档任务
  2. 验证两个用户的统计结果

- **预期结果**：两个用户都正确统计了对应年份表的流水数据，金额准确

### 1.2 归档任务异常场景测试

#### TC-008 用户无MongoDB流水时的归档处理
- **前置条件**：用户在MongoDB流水表中无记录，仅有MySQL流水数据
- **测试步骤**：
  1. 找一个只有MySQL流水的用户
  2. 执行归档任务
  3. 检查统计表该用户的记录
- **预期结果**：不生成统计记录或生成status=0的记录

#### TC-009 无法获取blackTime的用户归档处理
- **前置条件**：用户的5级降级查询都无法获取有效的blackTime
- **测试步骤**：
  1. 准备一个特殊用户数据
  2. 执行归档任务
  3. 检查该用户的统计结果
- **预期结果**：生成status=0的记录或跳过该用户

#### TC-010 本月已统计用户的跳过处理
- **前置条件**：用户本月已归档且statisticsDate为本月，归档任务不是本月第一次执行
- **测试步骤**：
  1. 第一次执行归档任务，记录某用户的updateTime
  2. 同月内第二次执行归档任务
  3. 检查该用户的updateTime是否发生变化
- **预期结果**：updateTime不变，表示跳过已统计用户

## 2. 路费欠款查询优化测试

### 2.1 查询优先级测试

#### TC-011 统计表有效时直接读取
- **前置条件**：用户在统计表有记录，status=1且statisticsDate为本月
- **测试步骤**：
  1. 在黑名单列表查看该用户的路费欠款
  2. 记录查询响应时间
  3. 查询统计表计算路费欠款：lateFee+lawyerFee+monthFee-lateFeeRefund
  4. 对比黑名单列表显示的金额
- **预期结果**：路费欠款与统计表计算结果一致，查询速度快（小于100ms）

#### TC-012 统计表无数据时降级实时聚合
- **前置条件**：用户在统计表无记录，但MongoDB流水表有数据
- **测试步骤**：
  1. 删除该用户的统计表记录
  2. 在黑名单列表查看该用户的路费欠款
  3. 记录查询响应时间
  4. 手工聚合MongoDB流水表验证金额
- **预期结果**：路费欠款正确显示（实时聚合结果），查询时间较长（大于500ms）

#### TC-013 统计表status=0时降级实时聚合
- **前置条件**：统计表有记录但status=0
- **测试步骤**：
  1. 修改统计表status为0
  2. 在黑名单列表查看该用户的路费欠款
  3. 验证金额准确性
- **预期结果**：降级实时聚合计算，金额准确

#### TC-014 统计表过期时的处理
- **前置条件**：统计表的statisticsDate不是本月
- **测试步骤**：
  1. 修改statisticsDate为上月
  2. 在黑名单列表查看该用户的路费欠款
  3. 验证查询结果
- **预期结果**：系统按设计处理，可能降级实时聚合或使用统计表历史数据

2. **查询优先级验证**：验证优先从统计表读取，统计表无效时降级实时聚合
3. **3个月判断逻辑**：验证系统正确判断3个月内外用户的处理逻辑

5. **性能提升验证**：验证查询性能提升达到预期（5倍以上）
