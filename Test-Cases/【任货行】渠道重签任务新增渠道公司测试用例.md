# 【任货行】渠道重签任务新增渠道公司测试用例

## 1. 新增任务弹窗功能测试

### 1.1 签订用户类型新增选项测试

#### TC-001 新增渠道公司选项UI显示测试
- **前置条件**：管理员已登录后台，进入渠道重签任务页面，点击“发起重签”
- **测试步骤**：
  1. 观察“新增”弹窗中的“签订用户类型”区域
  2. 检查是否存在“渠道公司”单选框
  3. 验证该选项与其他选项（所有用户、导入用户）的排版布局
- **预期结果**：
  1. “签订用户类型”中新增“渠道公司”单选框，显示正常
  2. 排版整齐，无样式错乱

#### TC-002 渠道公司选择交互测试
- **前置条件**：管理员在“发起重签”弹窗中选择了“渠道公司”作为签订用户类型
- **测试步骤**：
  1. 选中“渠道公司”单选框
  2. 观察界面变化，是否出现渠道公司选择器/搜索框
  3. 点击渠道公司选择器，尝试搜索并选择一个有效的渠道公司
  4. 验证“有效用户数”是否自动刷新为该渠道公司下的当前用户数
- **预期结果**：
  1. 选中后出现渠道公司选择组件
  2. 能正常搜索并选择渠道公司
  3. 选择后，“有效用户数”自动计算并显示该渠道下的需重签用户数量

### 1.2 创建渠道公司重签任务测试

#### TC-003 创建渠道公司重签任务流程测试
- **前置条件**：管理员已填写完成其他必填项（协议、版本、平台等），并在签订用户类型选择了“渠道公司”
- **测试步骤**：
  1. 选择一个具体的渠道公司（例如：渠道公司A）
  2. 点击“保存”或“确定”按钮
  3. 观察任务列表是否新增了一条任务
- **预期结果**：
  1. 任务创建成功
  2. 任务列表新增记录，状态为“签约中”（或初始状态）

## 2. 动态新增用户签约逻辑测试（核心需求）

### 2.1 小程序注册触发签约测试

#### TC-004 签约用户新增渠道公司-小程序注册场景
- **前置条件**：
  1. 已存在一个针对“渠道公司A”的进行中重签任务
  2. 待注册手机号未在系统中存在
- **测试步骤**：
  1. 使用新手机号在小程序端进行注册
  2. 注册时填写/绑定归属为“渠道公司A”
  3. 注册成功后，登录后台查看该重签任务的“待签约明细”
  4. 用户登录小程序，检查是否弹出签约弹窗
- **预期结果**：
  1. 后台“待签约明细”中自动新增该注册用户，且“渠道公司”列显示为“渠道公司A”
  2. 用户登录小程序后，自动弹出对应的协议重签/签约弹窗

### 2.2 后台创建用户触发签约测试

#### TC-005 签约用户新增渠道公司-后台创建场景
- **前置条件**：已存在一个针对“渠道公司A”的进行中重签任务
- **测试步骤**：
  1. 管理员在后台“货车用户管理”或相关模块创建一个新用户
  2. 将该用户归属设置为“渠道公司A”
  3. 保存并创建成功
  4. 查看重签任务的“待签约明细”
- **预期结果**：
  1. “待签约明细”中自动新增该用户记录
  2. 用户后续登录小程序时应触发签约流程

### 2.3 渠道公司新增用户触发签约测试

#### TC-006 渠道公司自行新增用户场景
- **前置条件**：已存在一个针对“渠道公司A”的进行中重签任务
- **测试步骤**：
  1. 登录“渠道公司A”的渠道后台（如有）或通过渠道接口/功能新增一个渠道用户
  2. 确保该用户成功归属到“渠道公司A”
  3. 检查管理后台重签任务的“待签约明细”
- **预期结果**：
  1. 新增的渠道用户自动加入到待签约明细列表中
  2. 该用户登录端需签署对应协议

## 3. 待签约明细功能测试

### 3.1 明细列表字段与检索测试

#### TC-007 待签约明细新增渠道公司字段验证
- **前置条件**：已创建包含渠道公司类型的重签任务，且有待签约用户数据
- **测试步骤**：
  1. 进入任务详情/待签约明细页面
  2. 浏览列表表头及数据列
  3. 检查是否存在“渠道公司”列
  4. 验证数据显示是否正确（显示用户所属的渠道公司名称）
- **预期结果**：
  1. 列表包含“渠道公司”列
  2. 数据显示准确，与用户实际归属一致

#### TC-008 渠道公司检索功能测试
- **前置条件**：待签约明细中包含多个不同渠道公司的用户（如支持多渠道选择场景，或验证全局检索）
  *注：如果任务是单选渠道公司，则验证该筛选条件是否默认选中或可用于验证数据*
- **测试步骤**：
  1. 在待签约明细页面的搜索/筛选区域
  2. 找到“渠道公司”检索项
  3. 输入或选择一个存在的渠道公司名称
  4. 点击查询
- **预期结果**：
  1. 列表仅显示归属于该渠道公司的用户数据
  2. 检索功能响应正常，数据准确

## 4. 数据一致性与边界测试

#### TC-009 任务取消后的新增用户处理
- **前置条件**：已存在一个针对“渠道公司A”的重签任务，但任务已被“取消”或“结束”
- **测试步骤**：
  1. 在“渠道公司A”下新增一个用户（注册或后台创建）
  2. 检查该用户是否被加入到已取消任务的明细中
  3. 用户登录小程序检查是否弹出该协议
- **预期结果**：
  1. 用户**不应**被加入到已结束/取消的任务明细中
  2. 用户登录**不应**弹出该任务对应的重签弹窗（除非是全局强制协议）

#### TC-010 变更用户渠道归属测试
- **前置条件**：
  1. 用户B归属于“渠道公司B”（无重签任务）
  2. “渠道公司A”有进行中的重签任务
- **测试步骤**：
  1. 将用户B的归属渠道变更为“渠道公司A”
  2. 检查重签任务明细
- **预期结果**：
  1. 根据需求定义，用户B**应**被追加到“渠道公司A”的重签任务明细中（需确认需求是否包含转入场景，按通常逻辑推断应包含）
  2. 用户B登录后需重签协议
