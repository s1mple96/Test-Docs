# 【任通行】无扣款方式下黑测试用例

## 1. 核心功能测试

### 1.1 下黑功能验证测试

#### TC-001 钱包2.0代扣无扣款方式下黑测试
- **前置条件**：用户钱包模式为MODE2（2），扣款方式为WITHHOLD（2），无有效银行卡，未开通微信代扣，用户状态为正常
- **测试步骤**：
  1. 准备符合条件的测试账户
  2. 手动执行定时任务ID 231
  3. 查询rtx_etc_card_user表中用户的status和black_resoure字段
  4. 查询rtx_black_record表中的下黑记录
  5. 查询小程序消息推送记录表
- **预期结果**：用户status更新为2（黑名单），black_resoure为7（无扣款方式），black_status为4（下黑失败，等待平台下黑），rtx_black_record表插入下黑记录，小程序消息内容为"账号无有效扣款方式，车辆已下黑"

#### TC-002 钱包1.0用户不受影响测试

  2. 手动执行定时任务ID 231
  3. 查询用户状态和black_resoure字段
  4. 确认用户是否被下黑
  5. 验证钱包1.0原有下黑解黑逻辑是否正常
- **预期结果**：用户状态不变，保持正常状态，不会被无扣款方式下黑任务处理，钱包1.0仍按原有逻辑处理（欠费自动下黑，余额充足自动解黑）

#### TC-003 钱包2.0余额模式用户不受影响测试

#### TC-004 已绑定有效银行卡用户排除测试

### 1.2 下黑优先级验证测试

#### TC-006 欠费下黑覆盖无扣款方式下黑测试
- **前置条件**：用户已被无扣款方式下黑（status为2，black_resoure为7），用户产生欠费账单
- **测试步骤**：
  1. 准备已无扣款方式下黑的测试账户
  2. 创建欠费账单数据
  3. 触发欠费下黑逻辑（钱包补扣失败场景）
  4. 查询用户的black_resoure字段
  5. 查询小程序消息推送记录
- **预期结果**：black_resoure从7更新为0（欠费），发送新的小程序消息"账号已欠费，车辆已下黑"，rtx_black_record表插入新的下黑记录，欠费下黑成功覆盖无扣款方式下黑

#### TC-007 欠费下黑用户不会被无扣款方式下黑覆盖测试
- **前置条件**：用户已被欠费下黑（status为2，black_resoure为0），用户无有效扣款方式
- **测试步骤**：
  1. 准备已欠费下黑的测试账户
  2. 确认用户同时满足无扣款方式下黑条件
  3. 手动执行定时任务ID 231
  4. 查询用户的black_resoure字段
  5. 验证是否有新的消息推送
- **预期结果**：black_resoure保持为0（欠费），不会更新为7（无扣款方式），不会发送新的小程序消息，定时任务检测到用户已是黑名单状态直接跳过

#### TC-008 无扣款方式下黑消息内容验证
- **前置条件**：用户满足无扣款方式下黑条件，driver_user_info表中有有效的wx_open_id
- **测试步骤**：

- **测试步骤**：

## 6. 异常场景测试

### 6.1 异常处理和容错测试

#### TC-020 分布式锁竞争测试
- **前置条件**：多个定时任务实例同时运行，存在相同的测试用户数据
- **测试步骤**：
  1. 准备测试数据
  2. 配置多个定时任务实例
  3. 同时触发多个实例执行定时任务ID 231
  4. 查看下黑结果和日志
  5. 验证是否存在重复下黑
- **预期结果**：同一用户只会被下黑一次，锁获取失败的任务记录日志并跳过该用户，无重复下黑，分布式锁机制正常工作

- **前置条件**：准备不同状态的测试用户
- **测试步骤**：
  1. 测试待激活用户（status为5）执行无扣款方式下黑
  2. 测试已注销用户（status为6）是否被下黑

  4. 查看任务执行日志
  5. 验证各状态用户的处理结果
- **预期结果**：待激活用户可以被下黑，已注销用户不会被下黑，未发行用户不会被下黑，系统正确识别和处理不同状态的用户

#### TC-022 数据库事务回滚测试
- **前置条件**：模拟数据库异常场景
- **测试步骤**：
  1. 在下黑过程中模拟数据库连接异常
  2. 查看事务是否正确回滚
  3. 验证用户状态数据一致性
  4. 查看rtx_black_record表是否有脏数据
  5. 验证系统异常处理机制
- **预期结果**：下黑失败时事务正确回滚，不会出现脏数据，用户状态保持一致，系统异常处理机制完善，日志记录完整
