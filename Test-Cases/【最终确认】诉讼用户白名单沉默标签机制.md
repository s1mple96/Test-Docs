# 【最终确认】诉讼用户白名单沉默标签机制

## 📅 确认日期
2025-11-12

## ✅ 开发已确认的完整逻辑

### 开发的原话

**测试人员问**：
> 是改为使用"白名单沉默标签"?

**开发回答**：
> 是的，起诉的时候会 先删除普通沉默标签，同时加上白名单沉默标签；这样后续就不会再打上沉默标签了，扣费就会扣不到了。

---

## 🎯 核心机制解析

### 完整的业务流程

```
用户被起诉时：
第1步：删除 Silent_Wallet（普通沉默标签）
第2步：添加"白名单沉默标签"（新标签）
       ↓
后续效果：
- 系统不会再给该用户打 Silent_Wallet 标签
- 扣款任务查询时，找不到 Silent_Wallet 标签
- 因此该用户不会被扣款
```

---

## 🔍 详细逻辑说明

### 1. 标签体系变化

#### 旧的设计（已废弃）

```
Litigation_Wallet（诉讼标签）❌ 不再使用
- 打标时机：用户起诉时
- 扣款逻辑：有该标签 → 跳过扣款
- 问题：需要在扣款时额外校验
```

#### 新的设计（当前方案）✅

```
白名单沉默标签（具体名称待确认）
- 打标时机：用户起诉时
- 删除标签：删除 Silent_Wallet
- 扣款逻辑：无需额外校验，自然扣不到
```

---

### 2. 为什么这样设计？

#### 原来的扣款逻辑

```
扣款条件：
1. ✅ 必须有 Silent_Wallet 标签
2. ❌ 不能有 MonthlyPayLaterComplaint_Wallet 标签
3. ❌ 不能有 Litigation_Wallet 标签 ← 需要专门校验
4. ✅ 冻结金额充足

问题：
- 每次扣款都要校验3个标签
- 逻辑复杂，容易遗漏
```

#### 新的扣款逻辑（更优雅）

```
扣款条件：
1. ✅ 必须有 Silent_Wallet 标签 ← 起诉用户没有这个标签
2. ❌ 不能有 MonthlyPayLaterComplaint_Wallet 标签
3. ✅ 冻结金额充足

优势：
- 起诉用户自然不满足第1条（因为 Silent_Wallet 已删除）
- 无需专门校验 Litigation_Wallet
- 逻辑简化，更不容易出错
```

---

### 3. "白名单沉默标签"的作用

#### 核心作用

```
作用1：防止再次打上 Silent_Wallet
- 系统定期给沉默用户打 Silent_Wallet 标签
- 如果只删除不保护，下次又会被打上
- 白名单沉默标签 = 保护标签，防止再次被打上

作用2：标识用户状态
- 标识该用户是"特殊的沉默用户"
- 与普通沉默用户区分
- 方便后续查询和统计
```

#### 为什么叫"白名单"？

```
白名单 = 豁免名单 = 保护名单

普通沉默用户：
- 有 Silent_Wallet 标签
- 会被扣款

白名单沉默用户：
- 有"白名单沉默标签"
- 不会有 Silent_Wallet 标签
- 在白名单保护中，不会被扣款
```

## 🎯 标签状态变化

### 普通沉默用户（会被扣款）

```
标签状态：
- ✅ 有 Silent_Wallet 标签
- ❌ 无 MonthlyPayLaterComplaint_Wallet 标签
- ❌ 无"白名单沉默标签"

扣款结果：✅ 会被扣款
```

### 起诉的沉默用户（不会被扣款）

```
起诉前：
- ✅ 有 Silent_Wallet 标签
- ❌ 无 MonthlyPayLaterComplaint_Wallet 标签

起诉后：
- ❌ 无 Silent_Wallet 标签（被删除）
- ✅ 有"白名单沉默标签"（新增）

扣款结果：❌ 不会被扣款（因为没有 Silent_Wallet）
```

### 客诉用户（不会被扣款）

```
标签状态：
- ✅ 有 Silent_Wallet 标签
- ✅ 有 MonthlyPayLaterComplaint_Wallet 标签

扣款结果：❌ 不会被扣款（因为有客诉标签）
```

## 🔧 技术实现要点

### 1. 起诉时的操作（原子性）

```java
@Transactional
public void userLitigation(String walletId) {
    // 第1步：删除普通沉默标签
    tagService.deleteTag(walletId, "Silent_Wallet");
    
    // 第2步：添加白名单沉默标签
    tagService.addTag(walletId, "WhiteList_Silent_Wallet"); // 标签名待确认
    
    // 第3步：记录起诉信息
    litigationService.recordLitigation(walletId);
    
    log.info("用户起诉处理完成：walletId={}", walletId);
}
```

### 2. 扣款任务（无需修改）

```java
// 扣款任务的查询条件（无需修改）
List<Bill> bills = billMapper.selectList(
    new QueryWrapper<Bill>()
        .eq("deduct_status", 1) // 待扣款
        .exists("SELECT 1 FROM user_wallet_tag uwt " +
                "WHERE uwt.wallet_id = monthly_fee_bill.wallet_id " +
                "AND uwt.tag_key = 'Silent_Wallet'") // 有沉默标签
        .notExists("SELECT 1 FROM user_wallet_tag uwt " +
                   "WHERE uwt.wallet_id = monthly_fee_bill.wallet_id " +
                   "AND uwt.tag_key = 'MonthlyPayLaterComplaint_Wallet'") // 无客诉标签
);

// 起诉用户自动被过滤（因为没有 Silent_Wallet 标签）
```

### 3. 防止再次打沉默标签

```java
// 沉默用户标签维护任务
public void maintainSilentWalletTags() {
    // 查询沉默用户（60天无通行记录）
    List<Wallet> silentWallets = walletService.getSilentWallets(60);
    
    for (Wallet wallet : silentWallets) {
        // 检查是否有白名单沉默标签
        boolean hasWhiteListTag = tagService.hasTag(
            wallet.getWalletId(), 
            "WhiteList_Silent_Wallet"
        );
        
        if (hasWhiteListTag) {
            // 有白名单标签，跳过，不再打普通沉默标签
            log.info("用户在白名单中，跳过：walletId={}", wallet.getWalletId());
            continue;
        }
        
        // 没有白名单标签，正常打普通沉默标签
        tagService.addTagIfNotExists(wallet.getWalletId(), "Silent_Wallet");
    }
}
```

## ✅ 结论

### 新机制的优势

```
1. ✅ 逻辑更简洁
   - 无需在扣款时专门校验诉讼标签
   - 自然过滤，不容易出错

2. ✅ 防止标签重复打上
   - 白名单标签保护用户
   - 不会再被打上普通沉默标签

3. ✅ 代码改动更小
   - 扣款任务无需修改
   - 只需修改起诉流程

4. ✅ 更容易理解和维护
   - 业务逻辑清晰
   - 标签职责明确
```
