# 【任通行】优化导入通行卡券的取数逻辑与校验测试用例

## 1. 需求背景分析

### 1.1 需求概述
- **需求方**：倪琦
- **需求背景**：目前逻辑为车辆ETC卡用户状态处于正常时，导入卡券才会成功，但是会遇到特殊情况，比如车辆提交了补换后的待激活状态，需优化
- **核心问题**：车辆在ETC卡用户有多条数据时取数的问题

### 1.2 现有代码逻辑分析
通过Arthas工具分析服务器上的实际代码（EtcUserCouponRecordController.issuanceCoupon方法），发现当前逻辑：

**当前代码逻辑**（第252行）：
```java
LambdaQueryWrapper qw = new LambdaQueryWrapper();
qw.eq(EtcCardUser::getCarNum, couponDataVo.getCarNum())
  .in(EtcCardUser::getStatus, Lists.newArrayList("1", "2", "3"));
qw.orderByDesc(BaseEntity::getCreateTime);
List userList = this.etcCardUserService.list(qw);
```

**现有逻辑问题**：
- 当前代码已经支持状态"1"（正常）、"2"（待激活）、"3"（已注销）
- 但需求要求排除"3"（已注销）状态
- 需求还要求排除"0"（未发行）、"4"（已注销已解绑）状态
- 当存在多条符合条件记录时，需要增加冲突检测逻辑

### 1.3 优化逻辑说明
**需要修改的逻辑**：
1. **状态过滤优化**：只允许状态"1"（正常）和"2"（待激活）的记录
2. **多条记录冲突检测**：若既存在正常记录又存在待激活记录，则报错
3. **排除不符合条件状态**：状态"0"（未发行）、"3"（已注销）、"4"（已注销已解绑）不予插入卡券

### 1.4 数据库状态码分析
基于rtx数据库rtx_etc_card_user表的status字段分析：
- status=0：未发行（508条记录）❌ 需排除
- status=1：正常（281条记录）✅ 允许导入  
- status=2：待激活（6条记录）✅ 允许导入（新增支持）
- status=3：已注销（29条记录）❌ 需排除（当前代码错误包含）
- status=4：已注销已解绑（165条记录）❌ 需排除
- status=5：未知状态（27条记录）❌ 需排除
- status=6：未知状态（36条记录）❌ 需排除
