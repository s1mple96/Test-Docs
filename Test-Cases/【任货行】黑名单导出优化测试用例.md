# 【任货行】黑名单导出优化测试用例

## 1. 数据归档功能测试

### 1.1 MongoDB流水数据归档测试

#### TC-001 用户最近下黑时间归档正常流程
- **前置条件**：MongoDB中存在流水数据,流水归档任务已完成
- **测试步骤**：
  1. 触发用户最近下黑时间归档任务
  2. 验证归档任务执行状态
  3. 查询归档后的用户下黑时间汇总表
  4. 对比原始流水数据和归档数据的准确性
- **预期结果**：归档任务执行成功,汇总表数据准确,与流水数据一致

#### TC-002 每月归档定时任务执行
- **前置条件**：已配置归档定时任务,流水归档已完成
- **测试步骤**：
  1. 等待归档定时任务触发时间点
  2. 检查任务是否自动执行
  3. 验证归档数据的完整性
  4. 检查归档执行日志
- **预期结果**：定时任务正常触发,归档数据完整,日志记录清晰

#### TC-003 归档数据更新时间校验
- **前置条件**：已有历史归档数据
- **测试步骤**：
  1. 记录当前归档表的更新时间
  2. 执行新一次归档任务
  3. 检查归档表更新时间是否变化
  4. 验证新归档数据是否覆盖旧数据
- **预期结果**：归档表更新时间正确更新,新数据成功覆盖旧数据

### 1.2 归档异常处理测试

#### TC-004 流水归档未完成时执行用户下黑时间归档
- **前置条件**：流水归档任务未完成或失败
- **测试步骤**：
  1. 确认流水归档状态为未完成
  2. 尝试执行用户下黑时间归档
  3. 检查系统处理逻辑
- **预期结果**：系统检测到流水归档未完成,提示错误或等待流水归档完成

#### TC-005 归档任务重复执行
- **前置条件**：归档任务正在执行中
- **测试步骤**：
  1. 启动第一次归档任务
  2. 在任务未完成时再次启动归档任务
  3. 观察系统处理逻辑
- **预期结果**：系统阻止重复执行或提示任务正在进行中

#### TC-006 归档数据为空的处理
- **前置条件**：MongoDB流水表中无数据
- **测试步骤**：
  1. 清空或使用空数据的测试环境
  2. 执行归档任务
  3. 检查归档结果和日志
- **预期结果**：任务正常完成,归档表为空或保留原数据,日志记录清晰

## 2. 下黑时间判断与查询优化测试

### 2.1 下黑时间边界判断测试

#### TC-007 下黑时间在数据库迁移时间内查询
- **前置条件**：用户下黑时间在数据库保留期内(未超过迁移时间)
- **测试步骤**：
  1. 选择下黑时间在近期的黑名单用户
  2. 执行欠费金额查询
  3. 验证查询数据来源(应仅查询数据库)
  4. 检查查询响应时间
- **预期结果**：仅查询数据库,不查询MongoDB,查询速度快

#### TC-008 下黑时间超过数据库迁移时间查询
- **前置条件**：用户下黑时间超过数据库迁移时间
- **测试步骤**：
  1. 选择下黑时间较早的黑名单用户
  2. 执行欠费金额查询
  3. 验证查询数据来源(应查询MongoDB)
  4. 检查查询响应时间和数据准确性
- **预期结果**：查询MongoDB获取历史流水数据,金额计算准确

#### TC-009 下黑时间等于迁移时间临界值
- **前置条件**：用户下黑时间等于数据库迁移时间点
- **测试步骤**：
  1. 准备下黑时间等于迁移时间临界点的测试数据
  2. 执行欠费金额查询
  3. 验证查询逻辑的边界处理
- **预期结果**：按预定边界规则查询,数据准确无遗漏

### 2.2 MongoDB金额统计优化测试

#### TC-010 按用户下黑时间统计MongoDB欠费金额
- **前置条件**：MongoDB中存在该用户的流水数据
- **测试步骤**：
  1. 选择有历史流水的用户
  2. 按用户下黑时间统计MongoDB中的欠费金额
  3. 验证统计金额的准确性
  4. 对比优化前后的统计时间
- **预期结果**：金额统计准确,查询时间显著减少

#### TC-011 用户无MongoDB流水数据的处理
- **前置条件**：用户下黑时间超过迁移时间,但MongoDB中无流水数据
- **测试步骤**：
  1. 选择无MongoDB流水的用户
  2. 执行欠费金额查询
  3. 检查系统处理逻辑
- **预期结果**：返回欠费金额为0或提示无流水数据,不报错

#### TC-012 MongoDB汇总表数据准确性验证
- **前置条件**：MongoDB汇总表已生成
- **测试步骤**：
  1. 随机选择10个用户
  2. 分别查询汇总表和原始流水表的欠费金额
  3. 对比两者数据是否一致
- **预期结果**：汇总表数据与原始流水数据完全一致

### 2.3 路费欠款查询逻辑优化测试

#### TC-013 优先从MongoDB汇总表获取欠款
- **前置条件**：MongoDB汇总表已生成,用户有历史欠款
- **测试步骤**：
  1. 执行路费欠款查询
  2. 监控查询路径(是否优先查询汇总表)
  3. 验证返回的欠款金额
  4. 对比查询性能(与直接查流水表)
- **预期结果**：优先查询汇总表,查询速度快,金额准确

#### TC-014 汇总表无数据时降级查询流水表
- **前置条件**：MongoDB汇总表暂无数据或数据未同步
- **测试步骤**：
  1. 清空或模拟汇总表无数据
  2. 执行路费欠款查询
  3. 验证是否降级查询原始流水表
- **预期结果**：自动降级查询流水表,保证查询正常,金额准确

#### TC-015 数据库与MongoDB数据组合查询
- **前置条件**：用户既有数据库流水又有MongoDB流水
- **测试步骤**：
  1. 选择跨越迁移时间点的用户
  2. 执行欠费金额查询
  3. 验证是否正确组合两部分数据
  4. 检查金额汇总的准确性
- **预期结果**：正确组合数据库和MongoDB数据,总欠款金额准确

## 4. 数据准确性测试

### 4.1 欠费金额准确性测试

#### TC-034 单一数据源欠款金额验证(仅数据库)
- **前置条件**：用户下黑时间在迁移时间内,流水仅在数据库
- **测试步骤**：
  1. 手工计算用户在数据库的欠费金额
  2. 系统查询该用户欠费金额
  3. 对比手工计算和系统查询结果
- **预期结果**：系统查询金额与手工计算一致

#### TC-035 单一数据源欠款金额验证(仅MongoDB)
- **前置条件**：用户下黑时间超过迁移时间,流水仅在MongoDB
- **测试步骤**：
  1. 手工统计MongoDB中该用户的流水金额
  2. 系统查询该用户欠费金额
  3. 对比手工统计和系统查询结果
- **预期结果**：系统查询金额与手工统计一致

#### TC-036 跨数据源欠款金额验证(数据库+MongoDB)
- **前置条件**：用户同时有数据库流水和MongoDB流水
- **测试步骤**：
  1. 分别统计数据库和MongoDB的流水金额
  2. 手工计算总欠费金额
  3. 系统查询该用户欠费金额
  4. 对比手工计算和系统查询结果
- **预期结果**：系统查询金额等于两部分金额之和,数据准确

#### TC-037 欠费金额包含优惠减免的处理
- **前置条件**：用户有优惠或减免记录
- **测试步骤**：

  2. 计算扣除优惠后的实际欠费
