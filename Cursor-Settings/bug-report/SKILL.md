# BUG 报告编写

## ⚠️ 核心原则

**测试工程师的职责是发现问题，开发工程师的职责是解决问题。**

- ✅ **只描述问题**: 清晰说明"发生了什么"、"如何复现"、"期望什么"
- ❌ **不分析代码**: 不定位代码位置、不分析根因、不给修复建议
- ❌ **不画蛇添足**: 只提供4个核心部分，其他由开发自己排查
- ✅ **只生成一个BUG报告文档**: 不生成其他辅助文档

## 📋 工作流程

```
[ ] 步骤1: 确认BUG现象
      ↓
[ ] 步骤2: 复现BUG（必须能稳定复现）
      ↓
[ ] 步骤3: 收集错误日志 ← 使用 ssh-operations skill
      ↓
[ ] 步骤4: 查询相关数据（如果涉及数据问题）← 使用 ssh-operations skill
      ↓
[ ] 步骤5: 编写BUG报告（只写4个核心部分）
```

**⚠️ 重要**:
- 步骤3和步骤4使用 **`ssh-operations` skill** 查询日志和数据库
- 步骤4查询数据时，只客观描述数据状态，不分析为什么
- 步骤5编写报告时，严格只包含4个部分，不添加其他内容

**📚 相关技能**:
- 使用 `ssh-operations` skill 查看日志、查询数据库
- 参考 [reference.md](reference.md) 获取系统路径、数据库配置

## BUG 报告模板

**⚠️ 严格只包含以下4个部分！**

```markdown
# BUG-【模块名】简短描述问题

## 1. BUG描述

问题现象的简洁描述，说明影响范围。

**影响**：
- ✅ 哪些功能正常
- ❌ 哪些功能异常

---

## 2. 复现步骤

### 前置条件
- 测试环境: 192.168.1.60
- 系统: 任通行/任货行/新发行
- 数据库: rtx/hcb
- 测试账号: xxx
- 测试数据: 关键ID（订单号/钱包ID/身份证号等）

### 操作步骤

**Step 1**: 第一步操作

```bash
# 附上完整的请求或命令
curl http://192.168.1.60:8080/api/xxx
```

**Step 2**: 查询相关数据（如果问题涉及数据）

```bash
# 数据查询
ssh_db_query(
  server="test",
  type="mysql",
  database="hcb",
  query="SELECT * FROM 表名 WHERE id='xxx'",
  dbUser="root",
  dbPassword="fm123456",
  dbHost="192.168.1.60",
  dbPort=3306
)
```

**数据查询结果**（只客观描述状态）:
```
# 订单存在 / 钱包不存在 / 字段值为NULL 等
字段1: 值1
字段2: NULL
```

**Step 3**: 观察结果

```bash
# 查看日志
ssh_execute(server="test", command="tail -n 100 /home/java-server/xxx/log.txt")
```

---

## 3. 预期结果

描述正确的行为应该是什么。

```
[INFO] - 应该看到的日志
[INFO] - 应该看到的输出
```

**预期**：
- ✅ 应该发生什么
- ✅ 应该看到什么数据状态

---

## 4. 实际结果

实际发生了什么，完整的错误日志。

```
[ERROR] - 完整的错误信息
java.lang.NullPointerException
    at xxx.xxx.xxx(xxx.java:123)
    ...（完整堆栈）
```

**实际**：
- ❌ 实际发生了什么
- ❌ 与预期的差异

---

**日志路径**: `/home/java-server/xxx/log.txt`
```

## 文件规范

- **位置**: `temp/BUG报告/`
- **命名**: `BUG-【模块】简短描述.md`
- **示例**: 
  - `BUG-【任通行】快速缴费充值失败.md`
  - `BUG-【任货行】黑名单导出缺少字段.md`

## ❌ 严格禁止的内容

以下内容**一律不要写**（这些是开发工程师的工作）:

1. ❌ **原因定位**: 不分析代码、不定位问题根因
2. ❌ **修复建议**: 不给出修复方案、不建议修改代码
3. ❌ **代码引用**: 不贴代码片段、不标注文件行号
4. ❌ **附件说明**: 日志已在"实际结果"中，不需要附件章节
5. ❌ **回归测试计划**: 测试计划自己管理，不写在BUG报告中
6. ❌ **影响评估**: 不评估资金损失、数据风险
7. ❌ **生产排查**: 不写排查SQL、修复脚本
8. ❌ **代码分析章节**: 如"根因分析"、"问题定位"、"技术分析"等

## 数据问题的处理方式

如果BUG涉及数据异常：

**✅ 正确做法**:
1. 提供数据查询SQL
2. 粘贴查询结果
3. **只客观描述数据状态**（存在/不存在/字段值）
4. 不分析为什么数据会这样

**示例**:

```markdown
**Step 2**: 查询问题数据

```bash
SELECT * FROM hcb_wallet WHERE id_code='142229198411114612'
```

**数据查询结果**:
```
Empty set (钱包不存在)
```
```

**❌ 错误做法**:
```markdown
**钱包不存在的原因**:  ❌ 不要写
因为注册流程没有创建钱包...
```

## 质量检查清单

编写BUG报告前检查:

- [ ] 只生成一个BUG报告文档，没有生成其他文档
- [ ] 文件保存到 `temp/BUG报告/` 目录
- [ ] 文件命名: `BUG-【模块】描述.md`
- [ ] 只包含4个核心部分（描述、复现、预期、实际）
- [ ] 没有"根因分析"、"修复建议"等禁止章节

- [ ] 错误日志完整粘贴

- [ ] 如涉及数据，只客观描述数据状态
- [ ] 没有分析为什么、没有给修复建议

## 常见错误（必须避免）

### 错误1: 添加"根因分析"章节

**错误**: 添加了代码分析、问题定位等章节  
**后果**: 越界、浪费时间、可能误导开发  
**正确**: 只写4个核心部分

### 错误2: 给出修复建议

**错误**: "建议修改xxx.java第123行"  
**后果**: 测试越界做开发的工作  
**正确**: 只描述问题，让开发自己分析

### 错误3: 分析数据为什么异常

**错误**: "钱包不存在是因为注册流程有问题..."  
**后果**: 这是猜测，可能不准确  
**正确**: 只说"钱包不存在"，不说为什么

### 错误4: 生成多个文档

**错误**: 生成BUG报告、代码分析、数据分析等多个文档  
**后果**: 文档太多，混乱  
**正确**: 只生成一个BUG报告文档
