# 测试用例编写

## ⚠️ 重要：必须遵循的工作流程

**编写测试用例前，必须按以下顺序执行所有步骤，不可跳过任何步骤！**

### 📋 工作流程清单（必须全部完成）

```
[ ] 步骤1: 分析需求文档 ← 最重要！不能漏测场景！
      ↓
[ ] 步骤2: 确定系统和数据库
      ↓
[ ] 步骤3: 拉取最新代码 (test分支)
      ↓
[ ] 步骤4: 分析代码逻辑（辅助理解）
      ↓
[ ] 步骤5: 查询数据库表结构（辅助理解）← 使用 ssh-operations skill
      ↓
[ ] 步骤6: 查询业务数据（辅助理解）← 使用 ssh-operations skill
      ↓
[ ] 步骤7: 提取测试场景和测试点
      ↓
[ ] 步骤8: 验证需求覆盖完整性（内部检查，不生成文档）
      ↓
[ ] 步骤9: 编写测试用例文档（只生成这一个文档！）
```

**⚠️ 核心原则**:
- **需求第一**: 测试用例必须覆盖所有需求场景，不能漏测！
- **代码为辅**: 代码分析是为了理解实现细节、补充测试点
- **防止漏测**: 每个需求点都必须有对应的测试用例
- **完整覆盖**: 必须检查需求覆盖率，确保没有遗漏
- **只生成一个文档**: 只输出测试用例文档，不生成其他辅助文档！

**📚 相关技能**:
- 使用 `ssh-operations` skill 查询数据库、查看日志、检查服务器文件
- 参考 [reference.md](reference.md) 获取系统路径、数据库配置

## 详细步骤说明

### 步骤1: 分析需求文档 🔥 最重要！必须执行

**这是最重要的步骤！测试用例的核心是覆盖需求，不能漏测！**

#### 需求来源

需求通常来自:
- PRD（产品需求文档）
- 用户故事
- 需求邮件/群消息
- JIRA/禅道等需求管理系统
- 口头需求（需让用户补充书面说明）

#### 需求分析方法

**1. 逐条提取需求点**

```markdown
示例需求: "新增快速缴费功能，用户可以选择充值金额，支持支付宝和微信支付"

提取需求点:
✓ 需求点1: 新增快速缴费功能
✓ 需求点2: 可以选择充值金额
✓ 需求点3: 支持支付宝支付
✓ 需求点4: 支持微信支付
```

**2. 识别需求场景**

每个需求点可能包含多个场景:
- 正常场景: 主流程、成功路径
- 异常场景: 失败路径、错误处理
- 边界场景: 最大值、最小值、临界值
- 特殊场景: 不同用户类型、不同权限、不同配置

**3. 提取业务规则**

注意需求中的:
- 金额范围: "0.01-50000元"
- 限制条件: "已实名用户才能充值"
- 时间限制: "每天最多充值3次"
- 状态流转: "待支付 → 支付中 → 支付成功"

**4. 识别隐含需求**

除了明确说明的需求，还要考虑:
- 权限控制: 谁可以操作
- 数据校验: 参数有效性
- 并发处理: 重复提交、并发扣款
- 异常处理: 网络超时、第三方异常
- 日志记录: 操作记录、审计日志

#### 需求覆盖检查清单

分析需求后，必须确认:

- [ ] 所有需求点已全部提取
- [ ] 每个需求点的所有场景已识别
- [ ] 业务规则已明确
- [ ] 隐含需求已考虑
- [ ] 没有遗漏的功能点

**⚠️ 防止漏测**: 如果需求不清晰，必须先找用户确认，不能凭猜测编写测试用例！

## 测试设计方法

### 等价类划分

将输入数据分为有效和无效类

**示例**: 充值金额(0.01-50000元)
- 有效: 100元、0.01元、50000元
- 无效: 0元、-100元、50001元、abc、空值

### 边界值分析

在边界处设计测试

**示例**:
- TC-001: 0.00元(最小-1) → 失败
- TC-002: 0.01元(最小) → 成功
- TC-003: 50000元(最大) → 成功
- TC-004: 50001元(最大+1) → 失败

### 场景法

根据业务流程设计

**主流程**: 登录 → 充值页 → 输入金额 → 支付 → 成功  
**异常流程**: 登录 → 充值页 → 输入金额 → 支付 → 失败 → 重试

### 错误推测

推测可能的错误: 空值、特殊字符、并发、重复提交、权限、数据一致性

## 特殊场景: 分页查询测试

**适用**: 列表查询、批量处理、定时任务

### 必测点

1. **代码检查**(优先): SQL是否有`ORDER BY`子句
2. **大数据量**: 准备数据>LIMIT值,验证无遗漏/重复
3. **稳定性**: 多次执行结果一致
4. **边界值**: 数据量=LIMIT值、LIMIT值+1

### 测试用例示例

```markdown
#### TC-010 代码检查-SQL是否有ORDER BY
- **前置条件**: 获取分页查询SQL代码
- **测试步骤**: 
  1. 查看Mapper.xml中的SQL
  2. 检查是否有ORDER BY子句
  3. 如缺少，提BUG
- **预期结果**: SQL包含`ORDER BY id ASC`

#### TC-011 大数据量-验证数据完整性
- **前置条件**: 准备5000条测试数据（LIMIT 2000）
- **测试步骤**: 
  1. 执行功能，记录处理的数据ID
  2. 对比是否所有数据都被处理
- **预期结果**: 5000条全部处理，无遗漏、无重复
```

**常见问题**: 缺少ORDER BY → 结果乱序 → 分页失效 → 数据遗漏/重复

## ❌ 常见错误（必须避免）

### 错误1: 跳过需求分析或需求分析不充分 🔥 最严重！

**错误做法**: 
- 没有仔细阅读需求文档
- 凭印象或猜测编写测试用例
- 需求不清晰时不去确认

**后果**: 
- **漏测需求场景** ← 最严重的问题！
- 测试用例与需求不符
- 需求覆盖不完整
- 上线后发现遗漏功能

**正确做法**: 
- 逐条提取需求点
- 识别所有需求场景（正常、异常、边界）
- 验证需求覆盖完整性
- 需求不清晰时必须找用户确认

### 错误2: 过度依赖代码分析，忽略需求

**错误做法**: 只看代码不看需求，按代码逻辑写测试用例

**后果**: 
- 代码实现可能与需求不符
- 只测了已实现的，遗漏了应该实现但未实现的
- 测试用例变成"测代码"而不是"测需求"

**正确做法**: 
- 以需求为主，代码为辅
- 先从需求提取测试场景
- 用代码分析补充技术细节

### 错误3: 跳过数据库查询步骤

**错误做法**: 不查询表结构和业务数据，凭想象写测试用例

**后果**:
- 字段名称错误
- 数据类型不匹配
- 遗漏必填字段
- 前置条件不准确

**正确做法**: 必须使用 `ssh_db_query` 工具查询表结构（DESC）和业务数据（SELECT）

### 错误4: 不验证需求覆盖完整性

**错误做法**: 提取测试点后直接写测试用例，不检查覆盖率

**后果**: 
- 遗漏部分需求场景
- 漏测关键功能
- 测试不充分

**正确做法**: 
- 列出需求点清单
- 每个需求点标注对应的测试用例编号
- 确认无遗漏

### 错误5: 使用错误的分支代码

**错误做法**: 拉取 master、dev 或其他分支的代码

**后果**: 代码与测试环境不一致，测试用例无法使用

**正确做法**: 只拉取 `test` 分支（测试环境分支）

### 错误6: 直接编写测试用例文档

**错误做法**: 一看到需求就直接写测试用例文档

**后果**: 测试用例完全不准确，需要返工重写

**正确做法**: 严格按照9步工作流程，完成前8步后才能写测试用例
