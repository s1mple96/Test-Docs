# 【任货行】非通行扣款账单空白属性优化 - 测试总结

**需求 ID**: 1017366  
**测试时间**: 2026-03-25  
**测试环境**: 192.168.1.60（`hcb` / Mongo `jdbdb`）

---

## 1. 需求概述

**需求内容**: 非通行扣款账单流水在部分场景下 `truckuser_id`、`bill_id` 及关联账单属性为空，导致后台导出列表大量空白。通过 Mongo 历史通行账单备份查询补全，并在扣款成功 MQ 链路中完善写入逻辑；另提供历史数据批量补充能力。

**涉及系统**

- 任货行定时任务（`hcbtimer`）：`DeductBillSuccessRabbitMqListener`、`HistoryArchivesService.findHisBy`、测试接口 `syncNonPassableDate`
- 任货行后台：非通行扣款账单列表/导出（`NonPassableDeductionBillFlowMapper.selectBillList`）

**涉及模块**

- 非通行扣款流水：`hcb_nonpassabledeductionbillflow`
- 历史归档：Mongo `hcb_hlj_heideducrecord_backup` 及按年集合（如 `_2024`、`_2025`）

---

## 2. 业务流程（摘要）

### 2.1 MQ 新数据（扣款成功）

1. 消费扣款成功消息，按 `deductBillId` + `operatorCode` 查 MySQL 通行账单；若无则按 **`HEIDEDUCRECORD_ID` + `OPERATOR_CODE`** 查 Mongo 备份（`findHisBy`，从当前年向下扫至 2024 及无后缀基表）。
2. 取 `service_fee`；仅当 **`service_fee` 非空且大于 0** 时组装并写入/更新非通行流水（含 `bill_id`、`fee_deduction_rules`、`bill_fee`、`deduction_rate` 等）。
3. 若 Mongo 无匹配记录，监听器打错误日志并 ack，不写入完整非通行扩展信息。

### 2.2 历史补数（`syncNonPassableDate`）

1. 查询待补集合：`deduction_id` 非空且 `truckuser_id`、`bill_id` 均为空，且 `deduction_time` 在入参时间范围内（`selectNeedSupplementListByDate`）。
2. 对每条用 `deduction_id` + `operator_code` 查 Mongo 备份；命中则回写 `truckuser_id`、`bill_id`（需两者在备份中均非空）。

### 2.3 后台导出

- 主表字段（账单 id、规则、金额、费率等）来自 `hcb_nonpassabledeductionbillflow`。
- 身份证、车牌、渠道等依赖 **`LEFT JOIN hcb_truckuser`**：`truckuser_id` 为空时，即使用例上「账单列」有值，用户维度列仍可能全空。

---

## 3. 业务逻辑（核心）

| 规则 | 说明 |
|------|------|
| Mongo 匹配键 | `HEIDEDUCRECORD_ID` = 流水的 `deduction_id`，且 `OPERATOR_CODE` 一致 |
| 历史扫描范围 | `findHisBy` 按年集合递减查询；无 `backup_2026` 等集合时，不表示逻辑错误，以是否归档为准 |
| MQ 写非通行条件 | `service_fee > 0` 才 `addOrNonPassableDeduction` |
| 补数范围 | 仅处理「双 ID 为空 + 时间窗内」的记录，已有值不覆盖 |

**数据表**

- `hcb_nonpassabledeductionbillflow`：非通行扣款流水
- Mongo：`HisArchivesEnum.HCB_HEIDEDUCRECORD_BACKUP` 对应集合及分年表

## 4. 测试过程

### 4.1 测试准备

- 用例文档：`temp/测试用例/【任货行】非通行扣款账单空白属性优化测试用例_V1.0.md`
- 覆盖：MQ 三月前/无备份、历史接口默认与自定义时间、已有数据不覆盖、Mongo 有/无备份、单条异常、导出与字段一致性、定时任务与 MQ 回归等（TC-001～TC-015）。

### 4.2 测试执行中的注意点

- 造数插入流水时若表对 `id` 无自增，需显式指定 `id`，避免 1364 错误。
- `deductionHisFailtTask` 等路径若依赖 `BILL_DATE`，造数需给合法日期，避免 `DateUtil` 空指针。
- 选取 Mongo 样本时，出口年需落在已有 `backup_*` 分表或基表中，否则 `findHisBy` 查不到。

### 4.3 导出验证结论

- **全量或不筛条件导出**：大量「账单 id / 规则 / 金额 / 费率」及用户列为空，多为历史未补数据或 `truckuser_id` 未关联，**不能单独作为需求失败依据**。
- **收窄条件后**：使用 **身份证 `441424199202126952`、车牌 `苏ZE5901`**（对应 `truckuser_id` 可查用户表）及窄扣款时间范围导出，**账单属性与用户相关列均有值**，与需求预期一致。

---

## 5. 关键发现

1. **导出空白分层**：需求主要解决 **流水表内 `bill_id` 等与 Mongo 补全**；身份证/车牌等空白常因 **`truckuser_id` 为空导致 JOIN 失败**，需区分展示。
2. **环境数据形态**：测试环境 Mongo 分年备份表是否齐全、归档是否到位，直接影响「能否补全」样本比例。
3. **验证方法**：以 **指定用户 + 时间范围** 或 **`bill_id`/`truckuser_id` 已填充子集** 导出，可有效避免脏数据干扰结论。

---

## 6. 测试结论

**整体评估**: ✅ 在收窄样本条件下，非通行扣款账单关键属性与后台导出表现符合需求；全量导出仍存在大量历史/不可补记录导致的空白，属预期数据形态，需在验收说明中与产品/业务对齐口径。

**是否可上线**: 以贵司发布流程为准；测试侧无额外阻塞项（基于当前对话与执行记录）。

**遗留/建议**

- 生产/预发关注：**历史补数任务是否按计划执行**、Mongo 归档与分表策略是否与 `findHisBy` 一致。
- 工单或运营若仍报「导出空」，优先核对：**该条 `deduction_id` 在 Mongo 是否有备份**、**`truckuser_id` 是否已写入**。
