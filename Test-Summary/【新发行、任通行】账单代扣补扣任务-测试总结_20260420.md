## 1. 需求概述

**需求内容**: 新增代扣补扣定时任务，针对首次扣款失败的账单在6-18点定期发起重试，同时对原有钱包补扣任务增加20分钟冷却过滤。

**涉及系统**:
- 任通行（RTX）：`java/rtx/`，数据库 `rtx`
- 新发行（fenmi-etc）：`java/fenmi/`，数据库 `fenmi_etc`

**涉及模块**:
- RTX：`BillDeductPayTask.failSupplementaryDeductTask`（新增）

- fenmi-etc：`BillWithholdDeductTask.failSupplementaryDeductTask`（新增）
- fenmi-etc：`BillWithholdDeductServiceImpl.failtBillDeductTask`（改动：新增20分钟过滤）

## 2. 业务流程

### 2.1 新增补扣任务流程

```
定时任务触发（6-18点，每2小时）
  → 查询：status=扣款失败 + create_time近3天 + deduc_time超过20分钟
  → 对每条账单发起补扣
  → 走用户配置的扣款顺序（银行卡 / 余额 / 微信代扣）
  → 更新账单状态（支付中→3 / 失败→2 / 成功→1）
```

**详细步骤**:
1. 按运营商+年份分批扫描扣款失败账单
2. 过滤条件：`status=2` + `create_time >= now()-3天` + `deduct_time <= now()-20分钟`
3. 对每条账单加分布式锁后发起扣款
4. RTX 钱包1.0模式：自动移除微信代扣方式（只走银行卡/余额）
5. RTX 钱包2.0模式：保留所有扣款方式
6. 补扣场景不触发扣款失败短信

### 2.2 原有钱包补扣任务改动

```
failtBillDeductTask / walletAdvanceDeductScene
  原来：扫描所有扣款失败账单
  现在：新增过滤 deduct_time <= now()-20分钟
```

目的：避免与 MQ 轮扣并发，轮扣进行中的账单（20分钟内有扣款动作）不再被补扣任务重复抓取。

### 2.3 任务执行时间控制

| 系统 | 控制方式 |
|------|---------|
| RTX | **代码内置**：`hour < 6 \|\| hour > 18` + Redis锁（TTL=2小时，key=`rtx:deduct:task:fail:`） |
| fenmi-etc | **外部Quartz cron**：`0 0 6,8,10,12,14,16,18 * * ?` |

## 5. 关键发现

### 5.1 存量问题（非本次引入）

| 问题 | 位置 | 描述 | 影响 |
|------|------|------|------|
| DateTimeParseException | RTX `BankCardBillDeductProcessor` | 账单 `bill_date='20260420'`（yyyyMMdd）被用 `yyyy-MM-dd` 格式解析失败 | 测试数据问题，生产数据是否有此格式需排查 |

### 5.2 测试发现的注意点

1. **fenmi-etc 使用 Sharding-JDBC 按运营商+年份分表**
   - 物理表：`etc_deduct_record_txb_2026`、`etc_deduct_record_mtk_2026` 等
   - 直接操作数据库时必须写入正确物理分表，否则任务扫描不到数据
   - RTX 同理，账单存于 `rtx_etc_deducrecord_txb`（逻辑表 `rtx_etc_deducrecord` 由 Sharding-JDBC 路由）

2. **fenmi-etc 测试用户支付通道问题**
   - 测试环境下，部分 fenmi-etc 用户支付通道（银行卡/微信）无有效配置
   - 导致首次扣款 `billDeduct` 所有方式均返回 ERROR（非 FAIL），账单状态卡在 status=0 而非 status=2
   - 解决：需确认测试用户在 `etc_wallet_deduction_method` 中有可用支付方式配置（如微信代扣授权）

3. **「近3天含当天」实现为72小时滑动窗口**
   - 代码：`LocalDateTime.now().minusDays(3)`
   - 非自然日（今天+昨天+前天从0点起算），建议与产品确认语义

4. **RTX任务配置问题**
   - 新任务（job_id=247）不能直接 SQL 插入 `sys_job` 后触发，需通过正式 API（POST /monitor/job）创建，否则 `qrtz_job_details` 不会同步，触发时报错 `The job referenced by the trigger does not exist`

5. **任通行 `system_job` 中 `job_group` 字段与 `qrtz_job_details` 中 `JOB_GROUP` 可能不一致**
   - 触发任务时需传正确的 `jobGroup`（需与 qrtz 表一致，不一定是 sys_job 中的值）

## 6. 注意事项

### 6.1 数据库操作注意

- ⚠️ fenmi-etc 账单表按 `operator_code + bill_year` 分表，读写必须指定正确物理表
- ⚠️ RTX 账单表按 `operator_code` 分表，物理表名如 `rtx_etc_deducrecord_txb`
- ⚠️ `bill_date` 格式在两系统有差异，fenmi-etc 用 `yyyy-MM-dd`，RTX 部分历史数据用 `yyyyMMdd`

### 6.2 测试环境注意

- RTX 计费任务（job 208）每次只处理部分账单，status=7 的账单可能需多触发几次才能完成计费
- fenmi-etc 首次扣款任务（job 44）cron=`0 0 1 * * ?`（凌晨1点），测试时需手动触发
- RTX 补扣任务（job 247）的 Redis 锁 key 为 `rtx:deduct:task:fail:`，TTL=2小时，同一时间窗口内只执行一次

### 6.3 后续测试建议

- 在真实生产账单上观察补扣任务运行情况，确认3天窗口语义（是否需要改为自然日）
- 关注补扣任务与23点钱包补扣任务（job 45）的执行时间是否有冲突（需求风险点3）
- 确认 RTX 日志中 `DateTimeParseException` 是否在生产环境中也会出现

## 7. 测试结论

**整体评估**: ✅ 通过

**是否可上线**: 是

**风险点**:
- 低风险：fenmi-etc 中「近3天」为72小时滑动窗口，与产品描述「含当天」可能有轻微语义差异，不影响主体功能
- 低风险：RTX 补扣任务与原有23点补扣任务的执行时间有轻微重叠风险（需求方已评估并提出6-18点窗口方案）

- fenmi-etc 测试用户（苏ZBV918/TXB）支付方式无有效配置，该账单无法完整走通补扣链路（测试数据问题，非代码问题）
